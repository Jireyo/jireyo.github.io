一、数据类型与结构

基本数据结构：String、List、Set、Zset、Hash
高级数据结构：Bitmap、Geo、HyberLogLog、Stream

redisObject

type和encoding共同实现了redis对象的多态处理。

对象共享
redis也有常量池，存了命令返回值（”OK”）和10000以内整数。

引用计数
refCount引用计数归零后可被回收

数据类型

String

编码
有三种编码类型：int、embstr、raw，其中字符串内容的字节超过一定阈值时使用raw。
int：long对象，与RedisObject一起分配内存。
embstr：SDS对象，与RedisObject一起分配内存，不可变。
raw：SDS对象，与RedisObject分开分配内存，可变。

应用
缓存对象，计数，分布式锁（SETNX实现「key不存在才插入」，同时设置过期时间）

SDS

简单动态字符串

组成
len(字符串长度)+alloc(字节数)+flags(类型标志)+buf[](数据)

优点

  • 常数复杂度获取字符串长度：只需要读取len属性即可了解SDS字符串长度，时间复杂度为O(1)。
  • 杜绝缓冲区溢出：修改字符串时，首先看len是否满足需求，不满足则进行可见扩展
  • 减少修改字符串的内存重新分配次数：对于修改字符串SDS实现了空间预分配和惰性空间释放两种策略：
    • 空间预分配：对字符串进行空间扩展的时候，扩展的内存比实际需要的多，这样可以减少连续执行字符串增长操作所需的内存重分配次数。
    • 惰性空间释放：缩短后多余的字节，而是使用 alloc 属性将这些字节的数量记录下来，等待后续使用。
  • 二进制安全：SDS不以空字符串判断结束，而是以len属性判断。

List

按插入顺序排序。

编码
List 类型的底层数据结构是由双向链表或压缩列表实现的，在 Redis 3.2 版本之后，List 数据类型底层数据结构就只由 quicklist 实现了。

应用
消息队列

双向链表

优点
在于访问/添加头节点尾节点快
缺点
在内存碎片多且浪费大

压缩列表


特点

  • 每个entry保存了本entry长度和前一个entry的长度，便于逆向搜索
  • 每个entry的类型根据实际长度灵活变化

优点
省空间

缺点
entry内容变化时，需要连锁更新

QuickList


双向链表 + 压缩列表（控制每个链表节点中的压缩列表的大小或者元素个数，来规避连锁更新的问题。）

Set

无序集合，成员唯一。

编码
Hash表或整数集合

应用
点赞、共同关注

Zset

权重排序

编码
压缩列表（小于128个元素，且元素小于64B）或跳表

应用
排行榜

Hash

编码
压缩列表或哈希表

应用
缓存对象

哈希表

rehash
条件：负载因子大于1且没有bgsave或bgrewriteaof任务，或负载因子大于5
创建一个2倍现在hash表大小的hash对象，同步迁移，完成后释放原对象。

渐进式rehash

  • 创建新hash对象；
  • 在 rehash 进行期间，每次哈希表元素进行新增、删除、查找或者更新操作时，Redis除了会执行对应的操作（查找时先找old再找new，新增时直接添加在new上）之外，还会顺序将「哈希表 1」中索引位置上的所有 key-value 迁移到「哈希表 2」上；
  • 随着处理客户端发起的哈希表操作请求数量越多，最终在某个时间点会把「哈希表 1 」的所有 key-value 迁移到「哈希表 2」，从而完成 rehash 操作。

应用
日活

Stream

消息队列

特点

  • 可自动生成不重复、递增的id。
  • 有消费组，组内消费者为竞争关系。
  • 每个消费组维护一个pending_ids，保证可靠性。只有消费者回复ACK后才将其标记为完成，消费者宕机重启后可查看未处理完的消息。
  • 其他消费组也可以将目标消费者超时未处理的消息转移到自己的pending列表
  • 死信，转发一定次数后都无法被消费者处理的消息，可输出XDEL。

与专业消息队列的差别

  • redis基于内存，宕机时磁盘中不是最新数据，主从切换时同理。专业消息队列基于集群，有多个副本。
  • redis基于内存，可能OOM，而Kafka和rabbitMQ基于磁盘

发布订阅方式为什么不能做消息队列

  • 不具有数据持久化
  • 不保存历史信息
  • 消息挤压时，消费者可能被强制断开

可结合MQTT、OPCUA的模式讲讲，发布订阅的目的是实时通讯，消息队列是为了可靠消费。

发布订阅

两种发布/订阅模式

  • 基于频道(Channel)的发布/订阅：频道以列表结构保存，发布的消息只发布给对于频道的消费者。
  • 基于模式(pattern)的发布/订阅：模式以链表结构保存，发布消息时，对每个模式进行表达式对比，然后选择发送。
