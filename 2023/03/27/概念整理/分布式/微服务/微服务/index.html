注册中心对比

实例类型
nacos有临时实例和非临时实例两种实例，其中临时实例在宕机超时后会被剔除。

心跳对比

  • Eureka：实例主动向注册中心发送心跳。
  • Nacos：临时实例主动向注册中心发送心跳，注册中心主动询问非临时实例的心跳

服务发现

  • Eureka：只支持拉取。
  • Nacos：支持拉取和注册中心主动推送，实现更新。

CAP

  • Eureka：AP模式（保证可用性）
  • Nacos：默认AP，集群中存在非临时实例时，使用CP（保证一致性）

对需要做负载均衡的服务（如本服务希望以指定的负载均衡方式调用userservice），则可以将本配置添加到本服务配置。

内置负载均衡规则类 规则描述
RoundRobinRule 简单轮询服务列表来选择服务器。它是Ribbon默认的负载均衡规则。
AvailabilityFilteringRule 对以下两种服务器进行忽略： （1）在默认情况下，这台服务器如果3次连接失败，这台服务器就会被设置为“短路”状态。短路状态将持续30秒，如果再次连接失败，短路的持续时间就会几何级地增加。 （2）并发数过高的服务器。如果一个服务器的并发连接数过高，配置了AvailabilityFilteringRule规则的客户端也会将其忽略。并发连接数的上限，可以由客户端的<clientName>.<clientConfigNameSpace>.ActiveConnectionsLimit属性进行配置。
WeightedResponseTimeRule 为每一个服务器赋予一个权重值。服务器响应时间越长，这个服务器的权重就越小。这个规则会随机选择服务器，这个权重值会影响服务器的选择。
ZoneAvoidanceRule 以区域可用的服务器为基础进行服务器的选择。使用Zone对服务器进行分类，这个Zone可以理解为一个机房、一个机架等。而后再对Zone内的多个服务做轮询。
BestAvailableRule 忽略那些短路的服务器，并选择并发数较低的服务器。
RandomRule 随机选择一个可用的服务器。
RetryRule 重试机制的选择逻辑

较好的策略是ZoneAvoidanceRule，即优先轮询同一个区域内的服务器。

最佳实践
FeignClient与被调用方的Controller很类似，希望能减少代码重复编写。

  • 继承方式：编写一个UserAPI接口，FeignClient和Controller分别继承该接口。
    • 特点：简单、实现了代码共享；耦合度高、且注解映射无法继承，Conrtoller必须再次声明发布方法、参数列表、注解。基本不考虑使用。
  • 抽取方式：将Feign的Client抽取到一个feign-api包中，所有微服务可引用该依赖包来实现对其他微服务的调用。
    • 特点：为所有微服务只需要编写一个feign-api包即可，实现所有微服务的Feign代码共享。
    • 使用：需要解决包扫描问题，引入feign-api包后，需要全局修改启动类@EnableFeignClients(basePackages = "cn.itcast.feign.clients")扫描包。

X、服务保护

概念

雪崩问题
某个服务出现故障，依赖该服务的的服务也会被阻塞，从而形成级联失败。

解决方案

  • 超时处理：请求超时没得到响应即返回，不会无休止等待。
  • 仓壁模式：业务线程限制，限制每个业务的最大线程数，避免影响其他业务。
  • 断路器：统计异常比例，超出阈值则熔断。
  • 限流（预防性措施）：为服务做限流处理。

流量控制

簇点链路
Controller -> Service ->Mapper调用链是一个簇点链路，每个节点都是一个资源。资源是流控、降级、热点参数限流、授权的基本控制单元。

流控阈值设置

  • QPS：限制每秒最大请求数
  • 线程数：限制业务最大线程数（需要配合FeighClient一起使用）

流控模式

  • 直接：根据自身的QPS或线程数控制流量
  • 关联：当关联资源触发阈值时，对当前资源限流
  • 链路：只有从指定入口进入当前资源的请求才会被限流。

其中：

  • 关联模式常用于两个竞争关系资源，可限制优先级，比如写业务频繁则限制读业务。
  • 链路模式用于存在调用关系的资源，即限制某个资源对当前资源的调用。

另外，Sentinel默认会给进入MVC的请求设置同一个资源，即将一个服务调用（Controller->Service->Mapper）视为一个资源节点，如果需要将Service类中的方法单独视为资源，应该：主动为其添加注解@SentinelResource(“resource_name”)

流控效果

  • 快速失败：达到阈值后直接返回flowexception。
  • warm up：预热模式，根据设置的预热时间，阈值从1/3最大阈值逐渐升高到最大阈值
  • 排队等待：超阈值请求排队等待，如果排队等待的预计等待时间（前面的请求数 × 平均请求处理时间）超过指定时间，则返回flowexception。

热点参数限流

  • 全局参数限流：设置参数索引、单机阈值、统计窗口。表示统计窗口内参数索引位置上相同的参数的请求不能超过阈值。
  • 热点参数限流：在全局参数限流的基础上，独立限制某个具体参数的请求值的阈值。

在热点参数限制中，MVC中所有资源无效，需要利用@SentinelResource注解标记资源。

熔断策略

  • 慢调用：统计时间内，时长超过最大RT的请求比例大于比例阈值，且总请求数大于最小请求数，则熔断指定时长。

原理

数据流
dashboard是sentinel控制台，core是微服务的sentinel依赖。

  • sentinel dashboard启动在指定的ip和port上；
  • sentinel core通过配置dashboard的ip和port与dashboard建立连接；
  • sentinel core通过HeartbeatSender向dashboard发送自己的存活信息，包括ip和port，dashboard将该信息保存；
  • sentinel core通过CommandCenter向dashboard提供数据请求服务，dashboard通过http向sentinel core请求数据。

实现原理

  • 每个资源都通过一个Entry对象维护资源的流量控制功能；
  • 整个调用链路是通过 Entry 的父子指针来描述的，SphU.Entry对象有如下属性：
    • 上下文对象context，贯穿整个请求的过程，保存于ThreadLocal中，每次调用链路开始都会创建一个context实例。保存有入口名称、调用根节点、调用源、当前执行点Entry。
    • 资源对象，包括资源名、入口/出口流量，资源类型。
    • ProcessSlot链，一个责任链，判断请求是否能否进入，通过SpiOrder可以设置各slot优先级。

  • entry.exit()：资源访问完毕后退出，为每个slot做一些资源清理和数据统计。

集群环境统计某一资源的所有访问量，Sentinel 中还对统计量进行了聚合，聚合后的统计节点会包含某一资源在不同入口下的所有访问量（称之为 ClusterNode），Sentinel 为了能够根据访问源进行流控还会针对每一个资源的每一类访问源分别进行流量统计（称之为 OriginNode）：


资源访问链

  • 资源的根节点为machine-root
  • ContextUtil.enter(resourceName,appname)表示调用进入该resourceName节点，该节点将被作为入口节点。
  • SphU.entry("nodeA")表示调用进入nodeA节点，并会经过该资源节点entry的slot链（entry中的clusterNode节点记录并提供资源的统计数据）。
