分支管理

checkout操作

  • 行为：
    • 对于工作区：如果做了修改但未暂存，则Git会检查更改与新分支上文件的冲突。
    • 对于暂存区：会随着分支切换出现在新分支的暂存区。
  • 处理：
    • 若需要更改到当前分支：需要先add，commit
    • 若暂时不想提交：需要使用git stash保存工作区和暂存区内容
    • 若要放弃更改：使用git checkout -- <file>放弃更改。

merge

merge操作
行为：

  • 对于工作区：将工作区、当前分支和merge分支的合并结果添加到工作区（包括冲突文件）
  • 对于暂存区：如果merge合并成功，则文件会被自动添加到暂存区并提交。若出现冲突，需要手动将冲突文件add进暂存区再手动commit。

处理：

  • merge行为会使用暂存区，因此最好先commit好暂存区再merge。
  • merge行为会将利用三个文件（工作区、版本库、merge分支）合并，因此如果工作区存在需要先添加到版本库的更改也要先进行add、commit。

模式：

  • Fast-forward模式（快进模式）：如果目标分支dev是本分支master的child（本分支在merge前相对dev没有其他提交），则本分支的指针会直接指向目标分支实现合并（以dev分支的最新提交作为本分支状态），这样的方式会屏蔽掉dev分支的改动细节，如下图所示。

  • No Fast-forward：当无法快进（即存在冲突）或强制禁用快进模式（使用–no-ff）时，会在merge时生成新的commit，如下图所示。

rebase

对比merge与rebase


rebase的场景一把是创建分支后，原分支有了一些新的提交，此时可以使用rebase将本分支的基更换为新提交，其优缺点如下：

  • 优点：让项目提交历史变得非常干净整洁，形成线性的项目提交历史。
  • 缺点：失去了安全性和可追溯性，因为这相当于对该分支的历史进行了修改，且其会为本分支的提交生成新的提交信息（commit id会变化）

rebase使用

rebase使用黄金法则
永远不要在公共分支上使用它，它会改变该分支的历史，导致其他使用该分支的用户出现历史错乱。

Force-Pushing
背景：如果确实对main分支进行了rebase操作，然后想把main分支推送到远程仓库。这时Git会因为本地分支的提交与远程分支的提交发生了冲突，而阻止这次的推送。
但是仍然可以通过使用–force选项来强行进行推送，这样会将远程仓库的历史更换为本地仓库的历史。
建议不要这样做，除非远程仓库的工作流确实需要被变更。此时仍要确认是否有其他也在这个分支上协作的用户。

远程分支

分支策略


在多人合作开发中，本地仓库分支状态：

  • master分支需要时刻与远程同步
  • dev分支也需要时刻与远程同步
  • bug分支无需推送到远程
  • feature分支根据需要推送到远程

工作流程

  • Clone下来时，本地只有master分支，需要使用git checkout -b dev origin/dev拉取远程的dev分支。
  • 正常情况，可以直接在本地修改dev分支后push到远端；但此时远端已经发生变换时，需要先pull合并解决dev分支的冲突后再push上去。

标签管理

管理规范

分支命名

  • master：主分支，部署生产环境
  • develop：开发分支，保持最新完成
  • feature：基于dev分支，命名为feature/module，module为开发的功能模块
  • test：测试环境，相对稳定
  • release：预发模块，由test或hotfix分支合并，不建议在release上修改代码
  • hotfix：以master为基线，创建hotfix分支及时修复，需要合并到master分支和develop分支

分支与环境

  • DEV 环境（Development environment）：用于开发者调试使用。
  • FAT环境（Feature Acceptance Test environment）：功能验收测试环境，用于测试环境下的软件测试者测试使用。
  • UAT环境 （User Acceptance Test environment）：用户验收测试环境，用于生产环境下的软件测试者测试使用。
  • PRO 环境（Production environment）：生产环境。

合并规范

Commit Message规范

简易版

标准版

  • type：提交类型
  • scope：可选项，本次 commit 波及的范围
  • subject：简明扼要的阐述下本次 commit 的主旨，在Angular Git Commit Guidelines中强调了三点。使用祈使句，首字母不要大写，结尾无需添加标点
  • body: 同样使用祈使句，在主体内容中我们需要把本次 commit 详细的描述一下，比如此次变更的动机
  • footer: 描述下与之关联的 issue 或 break change

type规范

  • feat: 新增功能
  • fix: 修复bug
  • docs: 仅文档更改
  • style: 不影响代码含义的更改（空白、格式设置、缺失 分号等）
  • refactor: 既不修复bug也不添加特性的代码更改
  • perf: 改进性能的代码更改
  • test: 添加缺少的测试或更正现有测试
  • chore: 对构建过程或辅助工具和库（如文档）的更改

此外还有：

  • delete：删除功能或文件
  • modify：修改功能
  • build：改变构建流程，新增依赖库、工具等（例如webpack、gulp、npm修改）
  • test：测试用例的新增、修改
  • ci：自动化流程配置修改
  • revert：回滚到上一个版本

单次提交注意事项

  • 提交问题必须为同一类别
  • 提交问题不要超过3个
  • 提交的commit发现不符合规范，需要通过进行以下的任一策略解决：
    • git commit --amend -m "新的提交信息"补充提交（该语句可以补充提交的文件内容，并修改message）。
    • git reset --hard HEAD 重新提交一次。
