微服务常见问题

一、RPC

RPC概念

什么是RPC？
Remote Procedure Calls 远程过程调用（RPC）是一种协议（规范），就是从一台机器（客户端）上通过参数传递的方式调用另一台机器（服务器）上的一个函数或方法（可以统称为服务）并得到返回的结果。
RPC的核心在于业务不需要了解底层网络技术的协议。在OSI网络通信模型中，其跨越了传输层和应用层。

RPC调用分类？

  • 同步调用
    • WebService：基于web容器，底层使用http协议，适合异构系统；
    • RMI：Remote Method Invocation远程方法调用，应用编程接口API，是Java对RPC的实现；
  • 异步调用
    • JMS：Java Message Service，主要实现为MQ。

RPC需要解决的问题

  • 通讯问题：需要通讯框架
  • 寻址问题：需要注册中心，服务发现
  • 序列化与反序列化问题：需要序列化协议

各功能具体实现

  • 通讯框架：
    • 大部分RPC框架都是基于TCP协议（应用层由语言自己实现，如java会基于netty开发对应的通讯框架）
    • 少量基于HTTP协议、UDP协议
  • 注册中心：
    • dubbo注册中心
    • eureka注册中心
    • zookeeper注册中心
    • 其他
  • 序列化协议：
    • Java序列化
    • Hession序列化
    • ProtoBuf序列化
    • Thrift序列化
    • 其他

应用级服务框架
应用级服务框架(RPC协议) 需要同时解决网络通讯、服务发现、序列化协议的问题，是对RPC的实现。

  • Dubbo
  • SpringCloud
  • gRPC
  • Thrift

RPC、RPC协议、RPC框架

  • RPC协议是对RPC的实现，而RPC框架是一种整体解决方案，由用户直接使用。
  • RPC协议的实现是不依赖于注册中心的，注册中心提供的是动态感知服务变化的能力。

gRPC和RPC的关系？
gRPC是一个高性能、通用的开源RPC框架，其由Google 2015年主要面向移动应用开发。需要定义proto文件供各端使用，其基于HTTP/2协议标准而设计（Java版本使用Netty库），性能更高。基于ProtoBuf序列化协议开发，二进制保证了更高的安全性。

使用的RPC框架是gRPC协议吗？
Triple 协议是 Dubbo3 设计的基于 HTTP 的 RPC 通信协议规范，它完全兼容 gRPC 协议，也是基于ProtoBuf序列化协议开发。

二、注册中心

各注册中心对比

注册中心对比和选型：Zookeeper、Eureka、Nacos、Consul和ETCD

Zookeeper的模式
推拉结合（只推送更新事件，由客户端拉取信息）

Zookeeper为什么不适用于注册中心？
当向注册中心查询服务列表时，我们可以容忍注册中心返回的是几分钟以前的注册信息，但不能接受服务直接down掉不可用。但Zookeeper选主时会出现不可用（强一致性），而作为注册中心，可用性的要求要高于一致性！

Eureka特点

  • 去中心化，不同于 ZooKeeper 的选举 leader 的过程，Eureka Server 采用的是Peer to Peer 对等通信。无 master/slave 之分，每一个 Peer 都是对等的，节点之间通过互相注册实现可用性，每个节点都可被视为其他节点的副本。
  • Eureka 为了保障注册中心的高可用性，容忍了数据的非强一致性，服务节点间的数据可能不一致， Client-Server 间的数据可能不一致。比较适合跨越多机房、对注册中心服务可用性要求较高的使用场景。

Nacos特点

  • Nacos是阿里开源的，支持基于 DNS 和基于 RPC 的服务发现。
  • Nacos 提供对服务的实时的健康检查，阻止向不健康的主机或服务实例发送请求。
  • 动态配置消除了配置变更时重新部署应用和服务的需要，让配置管理变得更加高效和敏捷。
  • Nacos的注册中心支持CP也支持AP
