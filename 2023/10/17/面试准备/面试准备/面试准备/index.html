为什么是偶发的？
大多数情况下，业务线程都在线程池操作该上下文变量前使用完毕并清除，但偶然的业务线程阻塞导致了该现象，导致业务无法进行，出现卡单。

如何排查

  1. 先回滚版本止血，暂时无问题
  2. 根据NPE的内容判断梳理链路，判断该上下文是否被正确的传入
  3. 检查发现无误后，模拟流量对故障机进行测试，发现无法复现问题
  4. 线上故障再次发生，集中团队成员进行Review，为系统方法设置更多的埋点，等待问题重现。

  6. 回滚置修改线程池前止血，对BUG进行修复，主要是对线程池的改造进行还原和对现有的所有业务中使用到ThreadLocal的业务代码安全性检验和排查。

技术点

OPCUA

定义

  • OPC（OLE for Process Control，用于过程控制的OLE）是一个工业标准，管理这个标准国际组织是OPC基金会。
  • OPC UA是一套通信协议标准，完成设备与设备，设备与应用之间的数据交互。它是一种应用层协议，基于tcp之上。

通讯模式
opc ua支持c/s模式，同时也支持类似mqtt的发布订阅模式。

通讯模型
OPC UA采用面向对象的设计思路， 使用了对象（objects）作为过程系统表示数据和活动的基础。对象包含了变量，事件和方法，它们通过引用（reference）来互相连接。
OPC UA 信息模型是节点的网络（Network of Node），或者称为结构化图（graph），由节点（node）和引用（References）组成，这种结构图称之为OPC UA 的地址空间。这种图形结构可以描述各种各样的结构化信息(对象)。

服务
服务可以看成是OPC UA服务器提供的API集合，OPC UA与定义了37个标准服务，常用的服务有：

  • 读写服务：可以获取和修改服务器指定节点指定属性的值
  • 调用服务：执行服务器上指定节点的方法
  • 订阅数据变化和订阅事件：可以监控服务器数据的变化

WebSocket

建立
WebSocket建立在一次HTTP请求后，在该HTTP请求中，加上Connection:Upgrade附带一个base64编码，服务端接收后对base64码进行公开算法加密并返回，客户端接收后也进行公开算法加密对比，相同则建立WebSocket连接。

特点
全双工、频繁交互（效率高）、实时响应。

网页刷新时会发生什么？

  • 前端：WebSocket默认会摧毁连接重连。
  • 后端：主要是清楚相关订阅信息
    • @OnClose处理关闭事件
    • @OnError处理错误事件

如何实现差异化推送？
算是实现了一个类似推模式的消息队列：

  • 建立连接后，客户端发送一个订阅报文，订阅数据项（做了一个Map<Integer,Set>，另一种方案是为每个数据项建立链表，实现更复杂，但不用遍历用户）。
  • 数据采集线程在处理数据时，会检查是否有用户订阅了该数据，然后发送。

故障处理

故障源

  • PLC：PLC的一些布尔数据作为故障源，服务器建立数据项与事件的映射表。
  • 系统：根据故障规则判断。

如何设置故障规则？

  • 单变量规则：
    • 事件变量（Boolean、Int）=> 事件，根据字典对应的事件。
    • 数值变量（Int、Double）=> 多级别阈值。
  • 多变量规则：
    • 建立规则表，一个规则对应一个规则表。
    • 本质上单变量规则表也是一个规则表中的一项，只不过后台自动填写了表达式，其作为独立规则作为单独字段保存，便于修改。
    • 多变量复杂，一个JS表达式关联一行数据，一个规则关联多个变量，一个变量也可以关联多个复杂规则。
    • JS表达式依赖：ScriptEngineManeger（内核为narshon或Rhino），可以通过每个产线数据的唯一变量名的方式编写函数，根据返回的Int值，判断报警等级。
  • 何时判断触发：
    • 有数据项变动时触发、定时任务触发（定时触发时需要主动读取）

规则保存在哪里，如何更新？
JVM内存，static CopyOnWriteHashMap

CopyOnWriteHashMap 和 ConcurrentHashMap对比

  • 前者适合读多写少场景，后者适合读少写多场景
  • CopyOnWrite读的时候是不需要加锁，写时先将容器复制一份，将新数据写在新容器中，再将原容器的引用指向新的容器，在读写的时候是加入重入锁。

故障后？
记录、推送、对是否触发密集采集做进一步判断。

密集采集？
规则会绑定一个是否触发采集的判断和采集的数据项集合、频率、次数，然后定时任务执行。

报警等级，行为，消警？
根据”0，1，2，3”等值，对同一个规则的警告进行调整，等级的变化都会产生一条新的报警明细记录报警时状态和时间戳。最后可以自动消除警告和手动消除（也插入一条数据）。
通过每条规则的最后一个明细的状态和时间戳判断，是否属于同一次报警。

如何防止重复报警？

  • 本次上报值>阈值 且上次上报值<=阈值，新开一次报警
  • 本次上报值>阈值 且上次上报值>阈值，不做变动，更新最后检测时间，如果发生了等级变化，则记录一条报警明细时间戳。
  • 规则同理，不过是根据规则返回值判断。

JVM调优

JVM调优实际案例

  • 配置JVM参数，如栈(-xss)、堆内存(-xms,-xmx)（IO密集型的可以稍微把「年轻代」空间加大些，因为大多数对象都是在年轻代就会灭亡。内存计算密集型的可以稍微把「老年代」空间加大些，对象存活时间会更长些）本应用中，就适当调大了young区，因为接收到的大多数产线数据处理后即回收。
  • -XX:MetaspaceSize，初始元空间大小，建议设置大一点，以免项目启动时为了调整元空间大小频繁Full GC。

技术点

MQTT

MQTT介绍
MQTT（Message Queuing Telemetry Transport，消息队列遥测传输协议），是一种基于发布/订阅（publish/subscribe）模式的“轻量级”通讯协议，该协议构建于TCP/IP协议上。作为一种低开销、低带宽占用的即时通讯协议，使其在物联网、小型设备、移动应用等方面有较广泛的应用。

MQTT优势

  • 轻量级：MQTT是二进制的协议，控制字段是精确到Bit级别的，其可以在严重受限的设备硬件和高延迟/带宽有限的网络上实现；
  • 灵活性：能够为IOT设备和服务多样化的应用场景提供支持的可能。

MQTT缺点

  • 不支持分包等机制：不适宜一些数据包特别大的应用场景。
  • 不支持负载分配：负载分配服务器对于高并发性和防止恶意攻击也是必不可少的。
  • 不支持脱机消息补偿：机器脱机后，不支持从MQTT服务器到设备的信息补偿。
  • 不支持点对点通信：从理论上讲，点对点通信可以通过相互订阅来实现，但是逻辑相对复杂并且涉及设备安全性。当设备B和设备C相同时-在主题的情况下，设备A无法知道消息是来自设备B还是来自设备C，并且消息很可能是被设备D窃听的。

MQTT为什么可以支持海量连接
MQTT是二进制协议，协议首部较小，因此很轻量级，服务器可以同时处理过多的消息收发。

MQTT工作模式
pub/sub，长连接，通过topic进行发布。

使用了什么中间件？最大连接数与QPS？如何测试

  • EMQ
  • 测试工具：XMeter，是JMeter在物联网MQTT测试下的扩展工具
  • 最大连接数：30000；（官方10W）
  • QPS：payload = 1KB下为10W左右（官方20W）
  • CPU：50-60%

服务质量？

  • QOS0：只发送一次（基于TCP/IP）
  • QOS1：至少发送一次，需要接收到返回值才停止。接收方可能重复接受（QoS 1 消息的重复在协议层面上是无法避免的。所以如果我们想要对 QoS 1 消息进行去重，只能从业务层面入手。需要做幂等）。
  • QOS2：确保只发送成功一次。进行两次往返，接收到第一次回复时丢弃包，并发送第二次确认消息发送成功的请求，收到第二次回复后，双方都确认了消息发送成功的事件。该模式下带有唯一消息 ID 的消息会存储两次，首先来自发送者，然后是接收者。

注意： QoS是Sender和Receiver之间的协议，而不是Publisher和Subscriber之间的协议。换句话说，Publisher发布了一条QoS1的消息，只能保证Broker能至少收到一次这个消息；而对于Subscriber能否至少收到一次这个消息，还要取决于Subscriber在Subscibe的时候和Broker协商的QoS等级。

性能差别
以 EMQX 为例，在相同的硬件配置下进行点对点通信，通常 QoS 0 与 QoS 1 能够达到的吞吐比较接近，不过 QoS 1 的 CPU 占用会略高于 QoS 0，负载较高时，QoS 1 的消息延迟也会进一步增加。而 QoS 2 能够达到的吞吐一般仅为 QoS 0、1 的一半左右。

设置为多少，为什么这么设置？
终端到broker的QOS和broker到服务器的设置均为1，由服务器实现幂等，处理方式是每一台设备的消息都有id号（或时间戳），这个号是自增的，监测到重复的消息时会忽略，也可以根据消息id是否连续判断是否有消息发送过丢失。
QoS 2 是最安全和最慢的服务质量级别，考虑网络状况，不适用该场景。

持久会话？
可以设置持久会话，Client掉线后重连，会重新获取消息（QOS1、QOS2）。

遗嘱机制？
遗嘱消息是 MQTT 为那些可能出现 意外断线 的设备提供的将 遗嘱 优雅地发送给第三方的能力。意外断线包括但不限于：

  • 因网络故障或网络波动，设备在保持连接周期内未能通讯，连接被服务端关闭
  • 设备意外掉电
  • 设备尝试进行不被允许的操作而被服务端关闭连接，例如订阅自身权限以外的主题等

遗嘱消息可以看作是一个简化版的 PUBLISH 消息，他也包含 Topic, Payload, QoS 等字段。遗嘱消息会在设备与服务端连接时，通过 CONNECT 报文指定，然后在设备意外断线时由服务端将该遗嘱消息发布到连接时指定的遗嘱主题（Will Topic）上。

与消息队列的对比？

  • 都是基于订阅发布模式
  • MQTT面向移动端，海量设备而消息少的情况；消息队列面向服务端，主要用于解耦、异步、削峰填谷。
  • MQTT没有持久化功能，主要用于实时通讯，消息队列会确保消息被消费才移除。
  • MQTT所有订阅者都可以获取订阅主题的消息（广播）
  • 消息队列需要显示的创建，MQTT的Topic随时创建。

组合使用

MQTT与OPCUA的区别和关系？

  • 相同点：
    • OPCUA和MQTT都是C/S架构，两者虽然协议内容不同，但最终目的一致，都是为了实现不同系统的数据交互；
  • 不同点：
    • 拓扑关系：
      • 在MQTT中，设备以及应用程序都作为客户端连接到MQTT服务器（Broker），形成以mqtt服务器为中心的拓扑结构；
      • 在OPC UA中，设备不是作为客户端，而是作为OPC UA Server提供各种服务，应用程序作为客户端连接到设备，调用OPC UA Server提供的各种服务。OPC UA中，设备除了作为OPC UA Server，也可以同时作为客户端连接到其他设备，形成非中心化的拓扑结构。
    • 业务模型：
      • MQTT标准中没有定义业务模型，MQTT服务器只是个中转站，用来透传转发多个客户端的消息。
      • OPC UA中，使用面向对象的方式定义业务模型。面向对象中通过类(包括类成员变量，类成员函数，类实例)，以及类之间的继承关系实现对业务的建模，而OPC UA的通过节点，以及节点之间的引用关系实现对业务的建模。客户端访问服务器提供的所有功能，都是通过调用服务完成。
    • 语义化对比：
      • MQTT标准中没有定义语义化的内容，应用程序要想知道设备提供了哪些功能/服务，只能靠应用程序和设备双方提前约定，或者通过阅读设备提供的说明文档。
      • OPC UA Server中节点的通用属性BrowseName、DisplayName、Description等描述了节点的属性和功能，应用程序连接到OPC UA Server后，通过浏览这些属性就可以了解设备提供的功能和服务。
  • 关联：可以结合OPCUA和MQTT协议，分别实现对设备数据的获取和对云端数据的上报。

MQTT安全机制
安全对于几乎所有的项目都是一个挑战，对于物联网项目更是如此：

  • 设备安全性与设备可用性之间往往是零和博弈。
  • 加密算法需要更多的计算能力，而物联网设备的性能往往非常有限。
  • 物联网的网络条件常常要比家庭或者办公室的网络条件差许多。

对于以上挑战，MQTT提供了多个层次的安全特性：

  1. 网络层：有条件可以通过拉专线或者使用VPN来连接设备与MQTT代理，以提高网络传输的安全性。
  2. 传输层：传输层使用TLS加密是确保安全的一个好手段，可以防止中间人攻击（Man-In-The-Middle Attack）。客户端证书不但可以作为设备的身份凭证，还可以用来验证设备。
  3. 应用层：MQTT还提供客户标识（Client Identifier）以及用户名密码，在应用层验证设备，还有IP过滤，ACL机制、防火墙来限制客户端访问。

认证与授权的区别：
在我们深入探讨访问控制列表（ACL）之前，有必要弄清楚授权和认证的区别。认证是指验证用户或系统的身份的过程。这通常是通过用户名和密码来实现的，但也可以采用生物识别或其他方式。而授权是指确定用户或系统可以执行哪些操作的过程。授权通常基于用户的角色或组成员身份。
ACL：规定了哪些客户端可以向哪些主题发布消息，以及哪些客户端可以订阅哪些主题。通过设置 ACL，MQTT Broker 可以保证只有经过授权的设备或客户端才能发布和订阅特定的主题，从而实现一个可控的通信环境。
