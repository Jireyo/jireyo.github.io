MySQL基础（查询与优化）

一、单表访问

访问方法（access method）

Mysql中查询的执行方式大致分两种：

  • 使用全表扫描进行查询
  • 使用索引进行查询（使用xx索引的xx查询）

访问方法分类：

  • all：全表
  • const：主键 = 常数
  • ref：二级索引 = 常数
  • ref_or_null：二级索引 = 常数 或 二级索引 = null
  • range：利用索引进行范围查询
  • index：不用回表的二级索引搜索
  • index merge：使用到多个索引来完成一次查询

range优化

  • 多range合并
  • 对于range and 其他条件，先根据range确定范围（将其他搜索条件替换为true，之后在筛选)。

index merge

  • Intersection合并，分别使用索引确定备选，然后取交集，出现以下情况才可能这样执行：
    • 情况一：二级索引列是等值匹配的情况。
    • 情况二：主键列可以是范围匹配
  • Union合并，分别使用索引确定备选，然后取并集，出现以下情况才可能这样执行：
    • 情况一：二级索引列是等值匹配的情况
    • 情况二：主键列可以是范围匹配
    • 情况三：与Intersection的筛选结果取并集
  • Sort-Union合并，两个二级索引搜索结果先按照主键值进行排序，之后按照Union索引合并方式执行的方式称之为Sort-Union索引合并。

注意：往往应该使用联合索引替代Intersection索引合并，即一个索引完成了两个列的范围搜索。

二、连接的原理

连接本质

连接的本质就是把各个连接表中的记录都取出来依次匹配的组合加入结果集并返回给用户，如下，这样的结果集称为笛卡尔积。

连接过程

驱动表
第一个需要查询的表，这个表称之为驱动表。
被驱动表
使用驱动表中的搜索结果，每条记录都对被驱动表进行一次搜索，过滤得到满足连接的结果。

eg:

两表连接查询中，驱动表只需要访问一次，被驱动表可能被访问多次。

内连接和外连接

  • 对于内连接的两个表，驱动表中的记录在被驱动表中找不到匹配的记录，该记录不会加入到最后的结果集。
  • 对于外连接的两个表，驱动表中的记录即使在被驱动表中没有匹配的记录，也仍然会加入到结果集。在MySQL中，根据选取驱动表的不同，外连接仍然可以细分为2种：
    • 左外连接：选取左侧的表为驱动表。
    • 右外连接：选取右侧的表为驱动表。

使用索引加快连接速度

加速连接查询主要是要加速被驱动表的查询。

  • 参考单表查询中索引的使用，建立合适的所以避免对被驱动表全表扫描。
  • 为了避免被驱动表反复被加载到内存中，使用join buffer：
    • 执行连接查询前申请一块固定大小的内存，先把若干条驱动表结果集中的记录装在这个join buffer中
    • 然后开始扫描被驱动表，每一条被驱动表的记录一次性和join buffer中的多条驱动表记录做匹配


加入了join buffer的嵌套循环连接算法称之为基于块的嵌套连接（Block Nested-Loop Join）算法。

驱动表的记录并不是所有列都会被放到join buffer中，只有查询列表中的列和过滤条件中的列才会被放到join buffer。

三、查询成本优化

成本

MySQL中一条查询语句的执行成本是由下边这两个方面组成的：

  • I/O成本
  • CPU成本

成本常数

  • 读取一个页面花费的成本默认是1.0。
  • 读取以及检测一条记录是否符合搜索条件的成本默认是0.2。

单表查询的成本

过程总结一下就是这样：

  1. 根据搜索条件，找出所有可能使用的索引
  2. 计算全表扫描的代价
  3. 计算使用不同索引（包括是否能使用联合索引）执行查询的代价
  4. 对比各种执行方案的代价，找出成本最低的那一个

全表扫描成本计算

由于查询成本=I/O成本+CPU成本，所以计算全表扫描的代价需要两个信息：

  • 聚簇索引占用的页面数
  • 该表中的记录数

索引查询成本的代价

范围搜索：

  • 查询优化器粗暴的认为读取索引的一个范围区间的I/O成本和读取一个页面是相同的。
  • 对于范围搜索如IN，会预估rows：
  • 先获取B+树的区间最左记录和区间最右记录，在统计两者之间的记录数量（index dive）
  • IN中参数过多（多于200）时，会使用基数估算索引。

使用索引处理对于range类型的查询条件

连接查询的成本

查询成本由下边两个部分构成：

  • 单次查询驱动表的成本
  • 多次查询被驱动表的成本

我们把对驱动表进行查询后得到的记录条数称之为驱动表的扇出
两种情况下计算驱动表扇出值时需要靠猜：

  • 全表扫描时，猜满足搜索条件的记录到底有多少条。
  • 索引扫描，猜满足除索引搜索条件外的其他搜索条件的记录有多少条。

两表连接的成本分析

内连接时，分析不同驱动表的情形，重点优化查询被驱动表的成本。

多表连接的成本分析

多表时，可能的搜索策略过多，MySQL有以下方法减少计算连接顺序的成本：

  • 提前结束某种顺序的成本评估：如果某次计算已经大于其他执行方式的成本，提前结束。
  • 系统变量optimizer_search_depth：连接表大于该值时，只对该值数量的表进行穷举分析。
  • 根据某些规则不考虑某些连接顺序：启发式规则，就是使用经验。

调节成本常数

  • 读取一个页面花费的成本默认是1.0
  • 检测一条记录是否符合搜索条件的成本默认是0.2

成本常数被存在mysql数据库中的server_cost表和engine_cost表中。

四、统计数据的收集

两种的统计数据存储方式

  • 永久性的统计数据：存硬盘
  • 非永久性的统计数据：存内存

InnoDB默认是以表为单位来收集和存储统计数据的，通过修改表可调整策略。

基于磁盘的永久性统计数据

统计数据存储到了两个表里，都位于mysql系统数据库，其中：

  • innodb_table_stats存储了关于表的统计数据，每一条记录对应着一个表的统计数据。
  • innodb_index_stats存储了关于索引的统计数据，每一条记录对应着一个索引的一个统计项的统计数据。

innodb_table_stats

字段名 描述
database_name 数据库名
table_name 表名
last_update 本条记录最后更新时间
n_rows 表中记录的条数
clustered_index_size 表的聚簇索引占用的页面数量
sum_of_other_index_sizes 表的其他索引占用的页面数量

n_rows是估计值

innodb_index_stats

字段名 描述
database_name 数据库名
table_name 表名
index_name 索引名
last_update 本条记录最后更新时间
stat_name 统计项的名称
stat_value 对应的统计项的值
sample_size 为生成统计数据而采样的页面数量
stat_description 对应的统计项的描述

定期更新统计数据

  • innodb_stats_auto_recalc：系统变量。默认为开启状态，当发送变动的记录超过表大小的10%，即重新计算。
  • ANALYZE TABLE：手动调用命令，重新计算。

基于内存的非永久性统计数据

非永久性的统计数据采样的页面数量是由innodb_stats_transient_sample_pages控制的，这个系统变量的默认值是8。

innodb_stats_method

是一个系统变量，设置对于允许NULL的索引列的基数计算策略：

  • nulls_equal：认为所有NULL值都是相等的。
  • nulls_unequal：认为所有NULL值都是不相等的。
  • nulls_ignored：直接把NULL值忽略掉。

五、基于规则优化

条件化简

  • 移除不必要的括号
  • 常量传递
  • 等值传递
  • 移除没用的条件
  • 表达式计算
  • HAVING子句和WHERE子句的合并：没有出现聚集函数时，合并HAVING子句和WHERE子句。
  • 常量表检测：表中只有一条数据或使用主键等值匹配的子句，优先执行。
  • 外连接消除：WHERE子句中指定被驱动表的 连接列 不为NULL，则变成了内连接。

子查询优化

子查询分类

按查询结果分类

  • 标量子查询
  • 行子查询
  • 列子查询
  • 表子查询

按与外层查询关系分类

  • 不相关子查询：子查询可以独立运行
  • 相关子查询：子查询的执行需要依赖于外层查询的值（一般表现为子查询中有带有外层表的条件）

子查询语法注意事项

  • 子查询必须用小括号扩起来。
  • 在SELECT子句中的子查询必须是标量子查询。
  • 对于[NOT] IN/ANY/SOME/ALL子查询来说，子查询中不允许有LIMIT语句。
  • 不允许在一条语句中增删改某个表的记录时同时还对该表进行子查询。

子查询在MySQL中执行

原始子查询：先执行子查询，然后将结果作为外层参数。

IN子查询优化（列子查询）

物化表

将不相关子查询的结果集写入一个临时表（基于内存的使用Memory存储引擎，并且会自动去重，如果太大了就会转而使用磁盘和B+树存储），这个表叫做物化表。

物化表转连接
将物化表转化成内连接，提供了转换驱动表的可能。

semi-join

对于s1表的某条记录来说，我们只关心在s2表中是否存在与之匹配的记录，而不关心具体有多少条记录与之匹配，最终的结果集中只保留s1表的记录。

内连接时，驱动表的每行都要在被驱动表中匹配（被驱动表中可能有多条匹配的行），但IN子句查询，只需要判断被驱动表中有匹配的行即可。

实现方式：

  • Table pullout （子查询中的表上拉）：连接列是被驱动表的唯一索引时，连接；
  • DuplicateWeedout execution strategy （重复值消除）：临时表去重；
  • LooseScan execution strategy （松散扫描）：子查询做驱动表，使用其结果扫描主表的各行结果是否满足条件；
  • Semi-join Materialization execution strategy：物化；
  • FirstMatch execution strategy （首次匹配）：原始查询。
小结
  • 满足转换为semi-join的条件时，查询优化器会优先把该子查询转换为semi-join，然后再考虑下边5种执行半连接的策略中哪个成本最低。
  • 不满足转换semi-join的条件时，查询优化器会从下边两种策略中找出一种成本更低的方式执行子查询：

    • 先将子查询物化之后再执行查询

    • 执行IN to EXISTS转换，转化后可能可以使用索引。

七、Buffer Poll

Buffer Poll概念

MySQL服务器启动的时候就向操作系统申请了一片连续的内存，叫做Buffer Pool（缓冲池）。

Buffer Pool大小

默认情况Buffer Pool只有128M，可以在启动服务器的时候配置innodb_buffer_pool_size参数的值来改变该值的大小。

Buffer Pool内部组成

Buffer Pool中默认的缓存页大小和在磁盘上默认的页大小是一样的，都是16KB。
每一个缓存页都有一个控制块，来装一些控制信息，这些控制信息包括该页所属的表空间编号、页号、缓存页在Buffer Pool中的地址、链表节点信息、一些锁信息以及LSN信息。

Buffer Poll管理

缓存页的哈希处理

Buffer Poll以表空间号 + 页号作为key，缓存页作为value创建了一个Hash表，如果访问时命中则直接访问，否则在free链表中申请。

free链表

把所有空闲的缓存页对应的控制块作为一个节点放到一个链表中，这个链表也可以被称作free链表

链表的基节点占用的内存空间并不包含在Buffer Pool中，而是另外单独申请的一块内存空间，后续的各种链表基节点同理。

flush链表

修改后的缓存页叫做脏页，为了避免频繁写磁盘影响性能，通过flush链表记录脏页，实现异步的写入磁盘。

LRU链表

Buffer Pool大小有限，有时需要将一些页淘汰出缓存。

简单的LRU链表

以按照最近最少使用的原则去淘汰缓存页，实现这个功能的链表被称为LRU链表（Least Recently Used）。
页面在被访问后就放在LRU链表的头部，淘汰时从尾部开始淘汰即可。

划分区域的LRU链表

存在的问题
  1. InnoDB对于可能会读取的页面，会进行预加载，也叫预读。
    • 线性预读：如果访问某个区的页面超过56，则异步读取下一个区所有页面。
    • 随机预读：如果缓存了某区的13个连续页面，则异步读取本区所有页面，默认关闭。

这会造成加载到Buffer Poll的页不一定用到，导致真正的热点数据淘汰。

  1. 全表扫描

非常多的使用频率偏低的页被同时加载到Buffer Pool时，可能会把那些使用频率非常高的页从Buffer Pool中淘汰掉。

解决方法

InnoDB把这个LRU链表按照一定比例分成两截，分别是：

  • 一部分链表叫做热数据，或者称young区域。
  • 另一部分叫做冷数据，或者称old区域。


默认情况下，old区域在LRU链表中所占的比例是37%。

通过分区和访问策略，分别解决了预读和全表扫描的问题。
访问策略

  • 页面初次加载时，对应控制块放在old区头部。
  • 在控制块上记录第一次访问时间，如果在较短时间（innodb_old_blocks_time = 1s）内再次访问，则不会将其移动到yound区头部。

优化LRU链表

频繁操作头节点开销较大，要降低调整LRU链表的频率。选择只有被访问的缓存页位于young区域的1/4的后边，才会被移动到LRU链表头部。

其他的一些链表

比如unzip LRU链表用于管理解压页，zip clean链表用于管理没有被解压的压缩页，zip free数组中每一个元素都代表一个链表，它们组成所谓的伙伴系统来为压缩页提供内存空间等等。

Buffer Poll中有：HashMap、FREE链表、LRU链表、FLUSH链表、其他链表。

空间释放

后台有专门的线程每隔一段时间负责把脏页刷新到磁盘，主要有两种刷新路径：

  • BUF_FLUSH_LRU：从LRU链表的冷数据中刷新一部分页面到磁盘
  • BUF_FLUSH_LIST：从flush链表中刷新一部分页面到磁盘

空间释放策略

  1. 当空间不够时，用户线程看LRU链表尾部有没有可以释放未修改页面
    • 有则释放
    • 没有则将LRU尾部脏页写盘后释放
  2. 特别繁忙时，用户线程批量从flush链表中将脏页刷入盘中。
