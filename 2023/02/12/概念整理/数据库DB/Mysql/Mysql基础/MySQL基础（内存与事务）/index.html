MySQL基础（内存与事务）

一、Buffer Poll

Buffer Poll概念

MySQL服务器启动的时候就向操作系统申请了一片连续的内存，叫做Buffer Pool（缓冲池）。

Buffer Pool大小

默认情况Buffer Pool只有128M，可以在启动服务器的时候配置innodb_buffer_pool_size参数的值来改变该值的大小。

Buffer Pool内部组成

Buffer Pool中默认的缓存页大小和在磁盘上默认的页大小是一样的，都是16KB。
每一个缓存页都有一个控制块，来装一些控制信息，这些控制信息包括该页所属的表空间编号、页号、缓存页在Buffer Pool中的地址、链表节点信息、一些锁信息以及LSN信息。

Buffer Poll管理

缓存页的哈希处理

Buffer Poll以表空间号 + 页号作为key，缓存页作为value创建了一个Hash表，如果访问时命中则直接访问，否则在free链表中申请。

free链表

把所有空闲的缓存页对应的控制块作为一个节点放到一个链表中，这个链表也可以被称作free链表

链表的基节点占用的内存空间并不包含在Buffer Pool中，而是另外单独申请的一块内存空间，后续的各种链表基节点同理。

flush链表

修改后的缓存页叫做脏页，为了避免频繁写磁盘影响性能，通过flush链表记录脏页，实现异步的写入磁盘。

LRU链表

Buffer Pool大小有限，有时需要将一些页淘汰出缓存。

简单的LRU链表

以按照最近最少使用的原则去淘汰缓存页，实现这个功能的链表被称为LRU链表（Least Recently Used）。
页面在被访问后就放在LRU链表的头部，淘汰时从尾部开始淘汰即可。

划分区域的LRU链表

存在的问题
  1. InnoDB对于可能会读取的页面，会进行预加载，也叫预读。
    • 线性预读：如果访问某个区的页面超过56，则异步读取下一个区所有页面。
    • 随机预读：如果缓存了某区的13个连续页面，则异步读取本区所有页面，默认关闭。

这会造成加载到Buffer Poll的页不一定用到，导致真正的热点数据淘汰。

  1. 全表扫描

非常多的使用频率偏低的页被同时加载到Buffer Pool时，可能会把那些使用频率非常高的页从Buffer Pool中淘汰掉。

解决方法

InnoDB把这个LRU链表按照一定比例分成两截，分别是：

  • 一部分链表叫做热数据，或者称young区域。
  • 另一部分叫做冷数据，或者称old区域。


默认情况下，old区域在LRU链表中所占的比例是37%。

通过分区和访问策略，分别解决了预读和全表扫描的问题。

访问策略

  • 页面初次加载时，对应控制块放在old区头部。
  • 在控制块上记录第一次访问时间，如果在特别短的时间（innodb_old_blocks_time = 1s）内再次访问，不会将其移动到young区头部。

优化LRU链表

频繁操作头节点开销较大，要降低调整LRU链表的频率。选择只有被访问的缓存页位于young区域的1/4的后边，才会被移动到LRU链表头部。

其他的一些链表

比如unzip LRU链表用于管理解压页，zip clean链表用于管理没有被解压的压缩页，zip free数组中每一个元素都代表一个链表，它们组成所谓的伙伴系统来为压缩页提供内存空间等等。

Buffer Poll中有：HashMap、FREE链表、LRU链表、FLUSH链表、其他链表。

空间释放

后台有专门的线程每隔一段时间负责把脏页刷新到磁盘，主要有两种刷新路径：

  • BUF_FLUSH_LRU：从LRU链表的冷数据中刷新一部分页面到磁盘
  • BUF_FLUSH_LIST：从flush链表中刷新一部分页面到磁盘

空间释放策略

  1. 当空间不够时，用户线程看LRU链表尾部有没有可以释放未修改页面
    • 有则释放
    • 没有则将LRU尾部脏页写盘后释放
  2. 特别繁忙时，用户线程批量从flush链表中将脏页刷入盘中。

  • 总大小 > 分片 * 每片大小，但不是后者的整数倍，则会调节成后者的整数倍。

二、事务

事务基本性质

  • 原子性（Atomicity）：事务是不可分割的操作
  • 一致性（Consistency）：保证数据库符合现实世界约束。
  • 隔离性（Isolation）：两个并发事务之间不能互相影响。
  • 持久性（Durability）：持久性意味着该转换对应的数据库操作所修改的数据都应该在磁盘上保留下来。

原子性和隔离性都是保证一致性的手段。

三、redo日志

redo日志简介

redo日志的目的是保证一致性（事务提交后一定执行）:

  • 若修改一个字段就对页进行一次刷盘，效率很低。
  • 事务可能涉及多个页，随机I/O效率低。

mysql通过redo日志记录事务执行的结果，并持久化到硬盘，相比将脏页持久化到硬盘:

  • 效率更高。
  • 顺序IO执行的。

redo日志格式

简单redo日志

只需要记录在某个页面修改/写入了几字节值类型的redo日志叫做简单redo日志，也叫物理日志：

  • MLOG_1/2/4/8BYTE（type字段对应的十进制数字为1/2/4/8）：表示在页面的某个偏移量处写入1/2/4/8个字节的redo日志类型。
  • MLOG_WRITE_STRING（type字段对应的十进制数字为30）：表示在页面的某个偏移量处写入一串数据，日志多了一个len字段。

简单redo日志只记录在哪个地方写入了什么数据。

复杂redo日志（插入日志）

插入新记录可能修改多个B+树，每个页面的统计信息也会更新：

解决方案

  • 方案一：在每个修改的地方都记录一条redo日志。
  • 方案二：将整个页面的第一个被修改的字节到最后一个修改的字节之间所有的数据当成是一条物理redo日志中的具体数据。

InnoDB的解决
提出了一些新的redo日志类型，比如：

  • MLOG_REC_INSERT（对应的十进制数字为9）：表示插入一条使用非紧凑行格式的记录时的redo日志类型。
  • MLOG_COMP_REC_INSERT（对应的十进制数字为38）：表示插入一条使用紧凑行格式的记录时的redo日志类型。
  • MLOG_COMP_LIST_START_DELETE（type字段对应的十进制数字为44）：表示从某条给定记录开始删除页面中的一系列使用紧凑行格式记录的redo日志类型。
  • 等等…

逻辑上看，这些复杂redo日志无法用来直接记录修改结果，而是将修改时的行为记录下来，恢复时需要调用相关函数进行恢复。

redo日志分组

实现分组

一条Sql语句可能产生多条redo日志，这些redo日志应该以组的形式写入磁盘，这是原子性的需要。

如何分组？

  • 每组的最后一条redo日志后边加上一条特殊类型的redo日志（分界符，只有一个type字段）
  • 对于只有一条redo日志的组，通过type的第一个bit位标识即可，避免浪费。

Mini-Transaction

对底层页面中的一次原子访问的过程称之为一个Mini-Transaction。

redo日志的写入

redo log block

通过mtr生成的redo日志都放在了大小为512字节的页中。这里的页与前面16KB的页不同，可以将其称为block加以区分。


LOG_BLOCK_HDR_NO是如何计算的
32位，前2位为0，所以该值最大为1G，意味着block块的个数不能超过1G。

redo log buffer

redo日志也不能直接直接写到磁盘上，服务器启动时就向操作系统申请了一大片称之为redo log buffer的连续内存空间。

通过启动参数innodb_log_buffer_size来指定log buffer的大小（默认16MB）。

redo日志写入log buffer

redo日志的写入是顺序的，InnoDB通过一个全局变量buf_free指明后续应该写入到哪个地方。

一个事务对应多个mtr，一个mtr对应一组redo日志，即一个事务有多个redo日志组。
由于事务是并发进行的，所以两个事务的redo日志组可能交替存储：

redo日志写入磁盘

在一些情况下redo日志会从log buffer中刷新到磁盘里，比如：

  • log buffer空间不足时
  • 事务提交时（持久性）
  • 将某个脏页刷新到磁盘前，会保证先将该脏页对应的 redo 日志刷新到磁盘中。
  • 后台线程
  • 正常关闭服务器时
  • 做checkpoint时

redo日志文件格式

redo日志文件组中的每个文件大小都一样，格式也一样，都是由两部分组成：

  • 前2048个字节，也就是前4个block是用来存储一些管理信息的。
  • 从第2048字节往后是用来存储log buffer中的block镜像的。

注意，这里都是以block作为单位，也就是一个日志存储单元的大小。

redo日志的id

log sequence number（LSN）

log sequence number是一个全局变量，初始值为8704。其帮助标识了每个mtr的redo日志组在log buffer中的起始和结束位置。

flushed_to_disk_lsn

redo日志是首先写到log buffer中，之后才会被刷新到磁盘上的redo日志文件。

  • log sequence number：全局变量，初始值为8704，标识mtr的redo日志组在log buffer中的位置。
  • flushed_to_disk_lsn：全局变量，表示刷新到磁盘中的redo日志量的全局变量。
  • buf_free：全局变量（指针），标志下一个mtr写入指针。
  • buf_next_to_write：全局变量（指针），标记当前log buffer中已经有哪些日志被刷新到磁盘中了。

总结：当有新的redo日志写入到log buffer时，首先lsn的值会增长，但flushed_to_disk_lsn不变，随后随着不断有log buffer中的日志被刷新到磁盘上，flushed_to_disk_lsn的值也跟着增长。

lsn值和redo日志文件偏移量的对应关系：
lsn的值是代表系统写入的redo日志量的一个总和，一个mtr中产生多少日志，lsn的值就增加多少（加上log block header和log block trailer的大小）

flush链表中的LSN

修改页内容后要将其控制块加入flush链表。

每个控制块记录两个属性：

  • oldest_modification：如果某个页面被加载到Buffer Pool后进行第一次修改，那么就将修改该页面的mtr开始时对应的lsn值写入这个属性。
  • newest_modification：每修改一次页面，都会将修改该页面的mtr结束时对应的lsn值写入这个属性。也就是说该属性表示页面最近一次修改后对应的系统lsn值。

eg:
修改页顺序：

  • mtr_1：a
  • mtr_2：b,c
  • mtr_3：b,d

一个mtr中修改的两个页的o_m和n_m字段是相同的，所以lsn的变动单位是一个mtr而不是一个redo日志！

redo日志的释放

checkpoint

redo日志文件大小有限，需要循环写入，因此redo日志文件需要释放空间：

  • 脏页刷盘前，redo日志必须写入磁盘；
  • 脏页刷盘后，redo日志可释放；

脏页刷盘、释放redo日志的过程本质是增加checkpoint_lsn的操作：

  • 计算一下当前系统中可以被覆盖的redo日志对应的lsn值最大是多少。
  • 将checkpoint_lsn和对应的redo日志文件组偏移量以及此次checkpint的编号写到日志文件的管理信息（就是checkpoint1或者checkpoint2）中。

checkpoint_lsn：脏页已持久化
flushed_to_dick_lsn：redo日志已持久化
lsn：当前redo日志序列

批量从flush链表中刷出脏页

当lsn增长过快时，系统无法及时做checkpoint，可能就需要用户线程同步的从flush链表中把那些最早修改的脏页（oldest_modification最小的脏页）刷新到磁盘。

innodb_flush_log_at_trx_commit的用法

如果对持久性要求不太高，可以修改innodb_flush_log_at_trx_commit系统变量

  • 0：当该系统变量值为0时，表示在事务提交时不立即向磁盘中同步redo日志；
  • 1：当该系统变量值为1时，表示在事务提交时需要将redo日志同步到磁盘；
  • 2：当该系统变量值为2时，表示在事务提交时需要将redo日志写到操作系统的缓冲区中；（这种情况下如果数据库挂了，操作系统没挂的话，事务的持久性还是可以保证的，但是操作系统也挂了的话，那就不能保证持久性了）。

崩溃恢复

恢复的起点
checkpoint_lsn之前的redo日志都可以被覆盖。

恢复的终点
log buffer文件是顺序IO的，找到没写满的block即可

怎么恢复

  • Hash：使用hash将同一个表中同一个页的redo日志放在一起处理。
  • 跳过已经刷新到磁盘的页面：checkpoint_lsn之后的redo日志可能也已经刷盘，根据每个页的File Header部分中的lsn字段可以判断本页最后一次刷盘时的lsn号，若其大于checkpoint_lsn，也不用恢复。

总结

  • redo日志的作用：记录事务执行的结果/过程，主要数据库崩溃时的恢复，实现持久性。
  • redo日志的结构：每条语句可能修改多个页面，对应着若干条redo日志记录，这些记录以组的形式（mtr）存在。
  • redo日志的存储：redo日志会先写入log buffer区，然后同步到系统上的log file中。
    • log buffer分若干个block块
    • 相比log buffer，log file的前4个块是固定的
  • redo日志的释放：脏页被刷盘时，形成一个check-point事件（记录当前check point lsn），该lsn之前redo日志不再被需要，可以释放。
  • 崩溃恢复时：从check point lsn开始恢复，遇到页的lsn记录大于check point lsn时略过。

四、undo日志

undo日志

redo是为了防止数据库崩溃（持久性），undo是为了防止事件回滚（原子性）。

事务id

当一个事务涉及到对某个表执行了增、删、改操作时，会为其分配一个事务id。
事务id由一个全局变量维护，新事务id自增1，当事务id达到256的倍数时，会将其记录下来Max Trx ID，下次启动时在该值基础上加256，防止事务id冲突。

undo日志的格式

undo日志是被记录到类型为FIL_PAGE_UNDO_LOG的页面中。
类似于redo日志：

  • 一个事务操作对应多个mtr，一个mtr对应多个undo日志。
  • undo no在一个事务中是从0开始递增的，也就是说只要事务没提交，每生成一条undo日志，那么该条日志的undo no就增1。
    • redo是以组mtr为单位记录，复写的；
    • undo日志是以事务为单位记录、还原的。
  • undo日志也有很多种类。

行记录

trx隐藏列

行格式中有一个trx_id字段，表示现在对该行进行修改的事务id。

roll_pointer隐藏列

指向记录对应的undo日志的一个指针。

undo日志类型

insert类


只需要记录插入id即可，回滚时根据id删除。

delete类

  • 在对一条记录进行delete mark操作前，需要把该记录的旧的trx_id和roll_pointer隐藏列的值都给记到对应的undo日志中来，形成版本链：


  • 额外记录删除时记录各列数据和索引各列信息。

删除页的过程：
1）将记录的delete_mask标识为改为1

2）删除事务提交后，有专门线程来将delete标识位为1的记录回收

update类

不更新主键
  • 如果更新后的列和更新前的列占用的存储空间都一样大，那么就可以进行就地更新。
  • 否则，用户线程直接先删除记录（记录加入垃圾链表），然后插入新记录。

更新主键
  • 将旧记录进行delete mark操作
  • 创建新记录并插入

对应主键更新，可以delete mask，否则必须先硬删除记录再插入新记录。

undo页面

FIL_PAGE_UNDO_LOG页面，该页面专门存储undo日志

  • TRX_UNDO_PAGE_TYPE：本页面准备存储什么种类的undo日志。
  • TRX_UNDO_PAGE_START：表示在当前页面第一条undo日志的起始偏移量。
  • TRX_UNDO_PAGE_FREE：最后一条undo日志结束时的偏移量。
  • TRX_UNDO_PAGE_NODE：代表一个List Node结构，连接前后的Undo页面（就像INODE页面一样）。

Undo页面链表

通过TRX_UNDO_PAGE_NODE形成链表。

一个undo页只记录一种类型的undo日志，不同类型的undo页形成链表：

针对普通表和临时表，又分别形成两组Undo日志。

这些链表是按需分配的。

undo链表是属于事务的：

每个事务分两组（普通/临时），每组有两条链表（update/insert），每个链表由若干该类型的undo页面连接构成。

undo日志具体写入过程

undo log segment header

每一个Undo链表都对应着一个段，称之为Undo Log Segment。

对于数据页，每个索引分两个段（叶子段/非叶子段），每个段有若干页面。
对于undo页，每个链表就是一个段。

  • TRX_UNDO_STATE：本Undo链表处在什么状态。
    • TRX_UNDO_ACTIVE：活跃状态，也就是一个活跃的事务正在往这个段里边写入undo日志。
    • TRX_UNDO_CACHED：被缓存的状态，等待被重用
    • TRX_UNDO_TO_FREE：insert undo链表不能被重用
    • TRX_UNDO_TO_PURGE：update undo链表不能被重用
    • 处于PREPARE阶段的事务产生的undo日志。
  • TRX_UNDO_LAST_LOG：本Undo页面链表中最后一个Undo Log Header的位置。
  • TRX_UNDO_FSEG_HEADER：本Undo页面链表对应的段的Segment Header信息
  • TRX_UNDO_PAGE_LIST：Undo页面链表的基节点。
  • undo page header 主要记录本页的指针信息
  • undo log segment 主要记录本undo log页链表的信息，便于快速搜索。
  • Segment Header 可以用来快速定位本段的INODE Entry（INODE Entry用来管理一个段中各页面的链表）。

undo log header

定义：同一个事务向一个Undo页面链表中写入的undo日志算是一个组，在undo log header记录这些组信息。

小结

  • 链表的第一个页面，会填充Undo Page Header、Undo Log Segment Header、Undo Log Header这3个部分，之后才开始正式写入undo日志。
  • 对于其他的页面只会填充Undo Page Header。

如前面总结，一个事务会持有多个undo log页面链表，在第一个链表中放入：

  • undo log segment header：记录本链表的状态（写入中？），并定位到INODE Entry，便于页面管理。
  • undo log header：记录本链表与事务相关的记录。

重用Undo页面

Undo页面链表是否可以被重用的条件很简单：

  • 该链表中只包含一个Undo页面。
  • 该Undo页面已经使用的空间小于整个页面空间的3/4：
    • insert undo链表可立刻删除。
    • update undo链表不能立刻删除。


回滚段

Rollback Segment Header页面

在这个页面中存放了各个Undo页面链表的frist undo page的页号，他们把这些页号称之为undo slot。

每个链表的头部记录了链表的状态信息和事务信息
这个页面中通过引用各个链表头部对链表进行统一的管理，方便复用：

  • 事务从回滚段中申请Undo链表。
  • 事务提交后，符合重用条件的可复用，不可重用的根据类型区分：
  • insert类型可释放
  • update类型需要放入history链表中。

多个回滚段

InnoDB中有127个回滚段，一个回滚段管理1024个undo链表。

回滚段的分类

  • 普通表回滚段
  • 临时表回滚段

划分原因：在修改针对普通表的回滚段中的Undo页面时，需要记录对应的redo日志，而修改针对临时表的回滚段中的Undo页面时，不需要记录对应的redo日志。

回滚段相关配置

  • 配置回滚段数量：启动参数innodb_rollback_segments
  • 通过innodb_undo_directory指定undo表空间所在的目录
  • 通过innodb_undo_tablespaces定义undo表空间的数量

总结

  • 一个事务对应多个mtr，一个mtr对应多个undo日志
  • undo日志存在undo页中，undo页分update和insert类。
  • 一个undo页不够记录一个事务的所有操作，所以要形成链表。每个事务持有2部分链表（普通/临时），1部分链表包括2条链表（update/insert）链表。（insert类型的undo页面在事务提交后就可以释放，update则不可以，这是处于版本链管理的需要）
  • 一个链表就是一个undo页面段，其也有一个INODE结构来管理这个段中各个页面?
  • 回滚段页面有各个链表头，实现对链表的管理。

MVCC原理

版本链

注意insert操作提交后对应的undo日志就没用了。

每次对记录进行改动，都会记录一条undo日志，每条undo日志也都有一个roll_pointer属性：

ReadView

对于使用READ COMMITTED和REPEATABLE READ隔离级别的事务来说：
必须保证可以读到已经提交了的事务修改过的记录。（未提交则不可见）

事务为了保证这种一致性，会在创建时，生成一个ReadView对象。

  • m_ids：表示在生成ReadView时当前系统中活跃的读写事务的事务id列表。
  • min_trx_id：表示在生成ReadView时当前系统中活跃的读写事务中最小的事务id，也就是m_ids中的最小值。
  • max_trx_id：表示生成ReadView时系统中应该分配给下一个事务的id值。
  • creator_trx_id：表示生成该ReadView的事务的事务id。

这样事务可以根据被访问版本的trx_id值判断：

  • 小于min_trx_id，说明trx_id事务已经被提交，可访问
  • 大于max_trx_id，说明trx_id事务在本事务开启后开启的，不可访问
  • 等于creator_trx_id，说明是本事务进行的修改，可访问
  • min与max之间，判断trx_id是否在m_ids中，在则说明开启本事务时trx_id事务未提交，不可访问，否则说明trx_id事务已经提交，可访问。

不同隔离级别的策略

  • READ COMMITTED —— 每次读取数据前都生成一个ReadView
  • REPEATABLE READ —— 在第一次读取数据时生成一个ReadView

六、锁

锁概念

锁其实是一个内存中的结构：

  • trx信息：代表这个锁结构是哪个事务生成的。
  • is_waiting：代表当前事务是否在等待。

MySQL在REPEATABLE READ隔离级别实际上就已经解决了幻读问题。

如何实现多隔离级别？

  • 方案一：读操作利用多版本并发控制（MVCC），写操作进行加锁。
  • 方案二：读、写操作都采用加锁的方式。

不同SQL操作对应的锁

  • 一致性读：也叫快照读，不加锁：SELECT ...
  • 锁定读：
    • 共享锁（S锁）：SELECT ... LOCK IN SHARE MODE;
    • 排他锁（X锁）：SELECT ... FOR UPDATE;
兼容性 X S
X 不兼容 不兼容
S 不兼容 兼容
  • 写操作：
    • DELETE：X锁
    • UPDATE：
      • 原地更新字段时（长度无变化）：X锁
      • 删除新增字段时（长度有变化）：先后执行DELETE和INSERT。
      • 更新主键时：先后执行DELETE和INSERT。
    • INSERT：不加锁，但使用隐式锁保证新增记录在提交前无法被访问。

多粒度锁

锁影响的范围称为锁的粒度，常见的有行级锁和表级锁。

意向锁：

  • 意向共享锁，英文名：Intention Shared Lock，简称IS锁。当事务准备在某条记录上加S锁时，需要先在表级别加一个IS锁。
  • 意向独占锁，英文名：Intention Exclusive Lock，简称IX锁。当事务准备在某条记录上加X锁时，需要先在表级别加一个IX锁。

意向锁：想加表锁的事务可以通过查看本表有没有意向锁判断是否有其他事务已经加了行锁。

IS锁和IX锁是兼容的：表示有事务在对表进行读、写（不是同一行）。

IX 不兼容 兼容 不兼容 兼容
S 不兼容 不兼容 兼容 兼容
IS 不兼容 兼容 兼容 兼容

MySQL中的行锁和表锁

其他存储引擎

对于MyISAM、MEMORY、MERGE这些存储引擎来说，它们只支持表级锁，也不支持事务。

InnoDB存储引擎中的锁

InnoDB中的表级锁

InnoDB存储引擎既支持表锁，也支持行锁：

  • 表级别的S锁、X锁
  • 表级别的IS锁、IX锁
  • 表级别的AUTO-INC锁：插入时上AUTO-INC锁，插入完成就释放

InnoDB中的行级锁

  • Record Locks：分S、X
  • Gap Locks：解决了幻读问题，加锁后其他事务无法在锁记录前面插入新记录，通过将锁放置在页面的最大记录上，其他事务就不能插入新记录了。


  • Next-Key Locks：相当于Record Locks+Gap Locks。

  • Insert Intention Locks：插入意向锁，事务在插入时被gap锁或者next-key lock阻塞，则生成一个Insert Intention Locks锁，主要起唤醒作用。


  • 隐式锁：如果某事务插入记录时，另一个事务要读取这个记录，会产生冲突（因为此时初始化未完成）。
    • 对于聚簇索引，另一个事务发现trx_id处于活跃状态时，会为插入事务创建一个X锁，并为自己创建一个X锁，形成等待。
    • 对于二级索引，通过Page Header中的PAGE_MAX_TRX_ID属性判断该页面的记录是否都是已提交状态（与当前活跃事务的最小id对比）。

InnoDB锁结构

  • 事务信息：事务的指针
  • 索引信息：加锁记录所属的索引
  • 表锁／行锁信息：
    • 标记锁了哪个表：SpaceID
    • 哪个表哪行记录：SpaceID,PageNumber,n_bits(一个数量，以便后面通过bit位标记页内哪些行加了锁)
  • type_mode：这是一个32位的数，被分成了lock_mode、lock_type和rec_lock_type三个部分，如图所示：
    • 锁的模式（lock_mode），占用低4位，可选的值如下：
      • LOCK_IS（十进制的0）：表示共享意向锁，也就是IS锁。
      • LOCK_IX（十进制的1）：表示独占意向锁，也就是IX锁。
      • LOCK_S（十进制的2）：表示共享锁，也就是S锁。
      • LOCK_X（十进制的3）：表示独占锁，也就是X锁。
      • LOCK_AUTO_INC（十进制的4）：表示AUTO-INC锁。
    • 锁的类型（lock_type），占用第5～8位，不过现阶段只有第5位和第6位被使用：
      • LOCK_TABLE（十进制的16），也就是当第5个比特位置为1时，表示表级锁。
      • LOCK_REC（十进制的32），也就是当第6个比特位置为1时，表示行级锁。
    • 行锁的具体类型（rec_lock_type），使用其余的位来表示。只有在lock_type的值为LOCK_REC时，也就是只有在该锁为行级锁时，才会被细分为更多的类型：
      • LOCK_ORDINARY（十进制的0）：表示next-key锁。
      • LOCK_GAP（十进制的512）：也就是当第10个比特位置为1时，表示gap锁。
      • LOCK_REC_NOT_GAP（十进制的1024）：也就是当第11个比特位置为1时，表示正经记录锁。
      • LOCK_INSERT_INTENTION（十进制的2048）：也就是当第12个比特位置为1时，表示插入意向锁。
      • LOCK_WAIT（十进制的256） ：也就是当第9个比特位置为1时，表示is_waiting为true，也就是当前事务尚未获取到锁，处在等待状态；当这个比特位为0时，表示is_waiting为false，也就是当前事务获取锁成功。

  • 其他信息：为了更好的管理系统运行过程中生成的各种锁结构而设计了各种哈希表和链表。
  • 一堆比特位：数量由行锁信息的n_bits属性决定，每bit与页内记录的heap_no对应
