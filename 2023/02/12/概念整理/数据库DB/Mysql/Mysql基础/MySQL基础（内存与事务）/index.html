MySQL基础（内存与事务）

一、Buffer Poll

Buffer Poll概念

MySQL服务器启动的时候就向操作系统申请了一片连续的内存，叫做Buffer Pool（缓冲池）。

Buffer Pool大小

默认情况Buffer Pool只有128M，可以在启动服务器的时候配置innodb_buffer_pool_size参数的值来改变该值的大小。

Buffer Pool内部组成

Buffer Pool中默认的缓存页大小和在磁盘上默认的页大小是一样的，都是16KB。
每一个缓存页都有一个控制块，来装一些控制信息，这些控制信息包括该页所属的表空间编号、页号、缓存页在Buffer Pool中的地址、链表节点信息、一些锁信息以及LSN信息。

Buffer Poll管理

缓存页的哈希处理

Buffer Poll以表空间号 + 页号作为key，缓存页作为value创建了一个Hash表，如果访问时命中则直接访问，否则在free链表中申请。

free链表

把所有空闲的缓存页对应的控制块作为一个节点放到一个链表中，这个链表也可以被称作free链表

链表的基节点占用的内存空间并不包含在Buffer Pool中，而是另外单独申请的一块内存空间，后续的各种链表基节点同理。

flush链表

修改后的缓存页叫做脏页，为了避免频繁写磁盘影响性能，通过flush链表记录脏页，实现异步的写入磁盘。

LRU链表

Buffer Pool大小有限，有时需要将一些页淘汰出缓存。

简单的LRU链表

以按照最近最少使用的原则去淘汰缓存页，实现这个功能的链表被称为LRU链表（Least Recently Used）。
页面在被访问后就放在LRU链表的头部，淘汰时从尾部开始淘汰即可。

划分区域的LRU链表

存在的问题
  1. InnoDB对于可能会读取的页面，会进行预加载，也叫预读。
    • 线性预读：如果访问某个区的页面超过56，则异步读取下一个区所有页面。
    • 随机预读：如果缓存了某区的13个连续页面，则异步读取本区所有页面，默认关闭。

这会造成加载到Buffer Poll的页不一定用到，导致真正的热点数据淘汰。

  1. 全表扫描

非常多的使用频率偏低的页被同时加载到Buffer Pool时，可能会把那些使用频率非常高的页从Buffer Pool中淘汰掉。

解决方法

InnoDB把这个LRU链表按照一定比例分成两截，分别是：

  • 一部分链表叫做热数据，或者称young区域。
  • 另一部分叫做冷数据，或者称old区域。


默认情况下，old区域在LRU链表中所占的比例是37%。

通过分区和访问策略，分别解决了预读和全表扫描的问题。

访问策略

  • 页面初次加载时，对应控制块放在old区头部。
  • 在控制块上记录第一次访问时间，如果在特别短的时间（innodb_old_blocks_time = 1s）内再次访问，不会将其移动到young区头部。

优化LRU链表

频繁操作头节点开销较大，要降低调整LRU链表的频率。选择只有被访问的缓存页位于young区域的1/4的后边，才会被移动到LRU链表头部。

其他的一些链表

比如unzip LRU链表用于管理解压页，zip clean链表用于管理没有被解压的压缩页，zip free数组中每一个元素都代表一个链表，它们组成所谓的伙伴系统来为压缩页提供内存空间等等。

Buffer Poll中有：HashMap、FREE链表、LRU链表、FLUSH链表、其他链表。

空间释放

后台有专门的线程每隔一段时间负责把脏页刷新到磁盘，主要有两种刷新路径：

  • BUF_FLUSH_LRU：从LRU链表的冷数据中刷新一部分页面到磁盘
  • BUF_FLUSH_LIST：从flush链表中刷新一部分页面到磁盘

空间释放策略

  1. 当空间不够时，用户线程看LRU链表尾部有没有可以释放未修改页面
    • 有则释放
    • 没有则将LRU尾部脏页写盘后释放
  2. 特别繁忙时，用户线程批量从flush链表中将脏页刷入盘中。

  • innodb_buffer_pool_size必须是innodb_buffer_pool_chunk_size × innodb_buffer_pool_instances的倍数，否则系统会自动调整innodb_buffer_pool_size的大小。

  • 总大小 < 分片 * 每片大小，则会调整每片大小
  • 总大小 > 分片 * 每片大小，但不是后者的整数倍，则会调节成后者的整数倍。

Change Buffer

Change Buffer概念

change buffer介绍
change buffer是内存中的一块区域，它保存在Innodb的buffer pool中，它在磁盘上也有对应的持久化空间，在系统表空间ibdata中，其默认最大只能占用到innodb buffer pool的25%大小。

change buffer作用
更新一条数据记录的时：

  • 如果记录所在的数据页在内存中，直接更新内存中的数据页A；
  • 如果记录所在的数据页不在内存中，Innodb会将这些更新操作缓存在change buffer中，当下次需要访问磁盘上的该数据页时，将数据页从磁盘上加载到内存里面，然后应用change buffer中与这个数据页有关的操作，即进行一次merge操作。

merge触发

  1. 访问数据页的时候
  2. 后台线程每秒都会merge
  3. 数据库正常关闭的情况下

change buffer优点

  1. 将原本2次的磁盘访问，整合成1次磁盘访问，并且能够保证数据的一致性。
  2. 数据页读入内存是需要占用内存空间的，这种方式能够避免内存的使用，提高内存的利用率。

普通索引和唯一索引

执行差别

  • 搜索时：唯一索引找到符合条件的结果即返回；普通索引则会接着向后寻找其他满足条件的结果。
  • 更新时：唯一索引的更新，不会使用到change buffer；普通索引的更新，会使用到change buffer。
    • 当要更新的记录在内存中的时候，两者差别不大：
      • 普通索引k会找到索引记录等于3和5的位置，然后在中间插入k=4的记录；
      • 唯一索引k会找到索引记录等于3和5的位置，然后判断没有冲突，插入k=4的记录；
    • 当要更新的记录不在内存中的时候：
      • 唯一索引需要将数据页加载到内存中，判断这个值没有冲突，然后插入这个新值；
      • 普通索引则是将更新记录在change buffer，语句执行就结束了。

普通索引利用了change buffer，减少了磁盘上的随机访问，对性能的提升比较明显。

区别原因
唯一索引在做insert或者update的时候，需要判断索引记录的唯一性（因为更新操作可能造成索引冲突），而判断唯一性必须要在内存中判断，所以数据页会被加载到内存中，如果数据页已经加载到了内存中，直接更新内存更快。

change buffer使用场景

写多读少的业务中，change buffer能够发挥很好的作用。它可以将多次对磁盘的操作，合并成一次merge操作，从而提高MySQL的性能，一次性merge的操作越多，收益就越大。但需要注意，如果你的数据写入之后。立马会读取(也就意味着需要从磁盘读取到内存)，那么建议不要使用change buffer，因为在这种情况下，使用change buffer不会减低IO次数，反而多了change buffer的维护开销。

change buffer和redo log的交互


当需要更新一条普通索引记录的时候：

  1. 如果这条记录在内存中，那么直接更新内存；
  2. 如果该记录没有在内存中，那么就需要更新change buffer
  3. 更新完change buffer之后，MySQL会在redo log中记录下change buffer的修改，
  4. 事务就算完成了，后续binlog落盘，redo log commit
  5. 当需要读取不在内存中的记录时，会将该数据页从磁盘加载到内存，然后应用change buffer中的修改，也就是merge操作

二、事务

事务基本性质

  • 原子性（Atomicity）：事务是不可分割的操作
  • 一致性（Consistency）：保证数据库符合现实世界约束。
  • 隔离性（Isolation）：两个并发事务之间不能互相影响。
  • 持久性（Durability）：持久性意味着该转换对应的数据库操作所修改的数据都应该在磁盘上保留下来。

ACID里的AID都是数据库的特征，也就是依赖数据库的具体实现。而唯独这个C,实际上它依赖于应用层，也就是依赖于开发者。这里的一致性是指系统从一个正确的状态迁移到另一个正确的状态。
即一致性是目的，AID是实现一致性的手段。

事务语法

事务五态模型

目前只有InnoDB和NDB存储引擎支持事务。

开启事务

  • SET autocommit=0：关闭自动提交（作用范围为session）
  • BEGIN
  • START TRANSACTION：后面可接修饰符指定事务模式（默认为读写模式）：
    • READ ONLY：只读事务
    • READ WRITE：读写事务
    • WITH CONSISTENT SNAPSHOT：启动一致性读，此时会立即建立本事务的一致性读ReadView，即开始事务了；

事务开始时点
通过语句SELECT * from information_schema.INNODB_TRX测试，执行start transaction后不会立即开启事务，只有实际上执行了业务语句时才会开启该事务。

autocommit和start transaction的使用

  • autocommit开启时：需要通过start transaction开启事务，此时自动提交将保持禁用状态，直到使用COMMIT或ROLLBACK结束事务。如果没有声明start transaction，则ROLLBACK语句是无效的（即使设置了save point）。
  • autocommit关闭时：可以不通过start transaction开启事务，也可以进行正常的提交和回滚。

三、redo日志

redo日志简介

redo日志的目的是保证一致性（事务提交后一定执行）:

  • 若修改一个字段就对页进行一次刷盘，效率很低。
  • 事务可能涉及多个页，随机I/O效率低。

mysql通过redo日志记录事务执行的结果，并持久化到硬盘，相比将脏页持久化到硬盘:

  • 效率更高。
  • 顺序IO执行的。

redo日志格式

简单redo日志

只需要记录在某个页面修改/写入了几字节值类型的redo日志叫做简单redo日志，也叫物理日志：

  • MLOG_1/2/4/8BYTE（type字段对应的十进制数字为1/2/4/8）：表示在页面的某个偏移量处写入1/2/4/8个字节的redo日志类型。
  • MLOG_WRITE_STRING（type字段对应的十进制数字为30）：表示在页面的某个偏移量处写入一串数据，日志多了一个len字段。

简单redo日志只记录在哪个地方写入了什么数据。

复杂redo日志（插入日志）

插入新记录可能修改多个B+树，每个页面的统计信息也会更新：

解决方案

  • 方案一：在每个修改的地方都记录一条redo日志。
  • 方案二：将整个页面的第一个被修改的字节到最后一个修改的字节之间所有的数据当成是一条物理redo日志中的具体数据。

InnoDB的解决
提出了一些新的redo日志类型，比如：

  • MLOG_REC_INSERT（对应的十进制数字为9）：表示插入一条使用非紧凑行格式的记录时的redo日志类型。
  • MLOG_COMP_REC_INSERT（对应的十进制数字为38）：表示插入一条使用紧凑行格式的记录时的redo日志类型。
  • MLOG_COMP_LIST_START_DELETE（type字段对应的十进制数字为44）：表示从某条给定记录开始删除页面中的一系列使用紧凑行格式记录的redo日志类型。
  • 等等…

逻辑上看，这些复杂redo日志无法用来直接记录修改结果，而是将修改时的行为记录下来，恢复时需要调用相关函数进行恢复。

redo日志分组

实现分组

一条Sql语句可能产生多条redo日志，这些redo日志应该以组的形式写入磁盘，这是原子性的需要。

如何分组？

  • 每组的最后一条redo日志后边加上一条特殊类型的redo日志（分界符，只有一个type字段）
  • 对于只有一条redo日志的组，通过type的第一个bit位标识即可，避免浪费。

Mini-Transaction

对底层页面中的一次原子访问的过程称之为一个Mini-Transaction。

redo日志的写入

redo log block

通过mtr生成的redo日志都放在了大小为512字节的页中。这里的页与前面16KB的页不同，可以将其称为block加以区分。


LOG_BLOCK_HDR_NO是如何计算的
32位，前2位为0，所以该值最大为1G，意味着block块的个数不能超过1G。

redo log buffer

redo日志也不能直接直接写到磁盘上，服务器启动时就向操作系统申请了一大片称之为redo log buffer的连续内存空间。

通过启动参数innodb_log_buffer_size来指定log buffer的大小（默认16MB）。

redo日志写入log buffer

redo日志的写入是顺序的，InnoDB通过一个全局变量buf_free指明后续应该写入到哪个地方。

一个事务对应多个mtr，一个mtr对应一组redo日志，即一个事务有多个redo日志组。

redo日志写入磁盘

在一些情况下redo日志会从log buffer中刷新到磁盘里，比如：

  • log buffer空间不足时
  • 事务提交时（持久性）
  • 将某个脏页刷新到磁盘前，会保证先将该脏页对应的 redo 日志刷新到磁盘中。
  • 后台线程
  • 正常关闭服务器时
  • 做checkpoint时

redo日志文件格式

redo日志文件组中的每个文件大小都一样，格式也一样，都是由两部分组成：

  • 前2048个字节，也就是前4个block是用来存储一些管理信息的。

redo日志的id

log sequence number（LSN）

log sequence number是一个全局变量，初始值为8704。其帮助标识了每个mtr的redo日志组在log buffer中的起始和结束位置。

flushed_to_disk_lsn

redo日志是首先写到log buffer中，之后才会被刷新到磁盘上的redo日志文件。

  • log sequence number：全局变量，初始值为8704，标识mtr的redo日志组在log buffer中的位置。
  • flushed_to_disk_lsn：全局变量，表示刷新到磁盘中的redo日志量的全局变量。
  • buf_free：全局变量（指针），标志下一个mtr写入指针。
  • buf_next_to_write：全局变量（指针），标记当前log buffer中已经有哪些日志被刷新到磁盘中了。

总结：当有新的redo日志写入到log buffer时，首先lsn的值会增长，但flushed_to_disk_lsn不变，随后随着不断有log buffer中的日志被刷新到磁盘上，flushed_to_disk_lsn的值也跟着增长。

lsn值和redo日志文件偏移量的对应关系：
lsn的值是代表系统写入的redo日志量的一个总和，一个mtr中产生多少日志，lsn的值就增加多少（加上log block header和log block trailer的大小）

flush链表中的LSN

修改页内容后要将其控制块加入flush链表。

每个控制块记录两个属性：

  • oldest_modification：如果某个页面被加载到Buffer Pool后进行第一次修改，那么就将修改该页面的mtr开始时对应的lsn值写入这个属性。
  • newest_modification：每修改一次页面，都会将修改该页面的mtr结束时对应的lsn值写入这个属性。也就是说该属性表示页面最近一次修改后对应的系统lsn值。

eg:
修改页顺序：

  • mtr_1：a
  • mtr_2：b,c
  • mtr_3：b,d

一个mtr中修改的两个页的o_m和n_m字段是相同的，所以lsn的变动单位是一个mtr而不是一个redo日志！

redo日志的释放

checkpoint

redo日志文件大小有限，需要循环写入，因此redo日志文件需要释放空间：

  • 脏页刷盘前，redo日志必须写入磁盘；
  • 脏页刷盘后，redo日志可释放；

脏页刷盘、释放redo日志的过程本质是增加checkpoint_lsn的操作：

  • 计算一下当前系统中可以被覆盖的redo日志对应的lsn值最大是多少。
  • 将checkpoint_lsn和对应的redo日志文件组偏移量以及此次checkpint的编号写到日志文件的管理信息（就是checkpoint1或者checkpoint2）中。

checkpoint_lsn：脏页已持久化
flushed_to_dick_lsn：redo日志已持久化
lsn：当前redo日志序列

批量从flush链表中刷出脏页

当lsn增长过快时，系统无法及时做checkpoint，可能就需要用户线程同步的从flush链表中把那些最早修改的脏页（oldest_modification最小的脏页）刷新到磁盘。

innodb_flush_log_at_trx_commit的用法

如果对持久性要求不太高，可以修改innodb_flush_log_at_trx_commit系统变量

  • 0：当该系统变量值为0时，表示在事务提交时不立即向磁盘中同步redo日志；
  • 1：当该系统变量值为1时，表示在事务提交时需要将redo日志同步到磁盘；
  • 2：当该系统变量值为2时，表示在事务提交时需要将redo日志写到操作系统的缓冲区中；（这种情况下如果数据库挂了，操作系统没挂的话，事务的持久性还是可以保证的，但是操作系统也挂了的话，那就不能保证持久性了）。

崩溃恢复

恢复的起点
checkpoint_lsn之前的redo日志都可以被覆盖。

恢复的终点
log buffer文件是顺序IO的，找到没写满的block即可

怎么恢复

  • Hash：使用hash将同一个表中同一个页的redo日志放在一起处理。
  • 跳过已经刷新到磁盘的页面：checkpoint_lsn之后的redo日志可能也已经刷盘，根据每个页的File Header部分中的lsn字段可以判断本页最后一次刷盘时的lsn号，若其大于checkpoint_lsn，也不用恢复。

总结

  • redo日志的作用：记录事务执行的结果/过程，主要数据库崩溃时的恢复，实现持久性。
  • redo日志的结构：每条语句可能修改多个页面，对应着若干条redo日志记录，这些记录以组的形式（mtr）存在。
  • redo日志的存储：redo日志会先写入log buffer区，然后同步到系统上的log file中。
    • log buffer分若干个block块
    • 相比log buffer，log file的前4个块是固定的
  • redo日志的释放：脏页被刷盘时，形成一个check-point事件（记录当前check point lsn），该lsn之前redo日志不再被需要，可以释放。
  • 崩溃恢复时：从check point lsn开始恢复，遇到页的lsn记录大于check point lsn时略过。

四、undo日志

undo日志

redo是为了防止数据库崩溃（持久性），undo是为了防止事件回滚（原子性）。

事务id

当一个事务涉及到对某个表执行了增、删、改操作时，会为其分配一个事务id。
事务id由一个全局变量维护，新事务id自增1，当事务id达到256的倍数时，会将其记录下来Max Trx ID，下次启动时在该值基础上加256，防止事务id冲突。

undo日志的格式

undo日志是被记录到类型为FIL_PAGE_UNDO_LOG的页面中。
类似于redo日志：

  • 一个事务操作对应多个mtr，一个mtr对应多个undo日志。
  • undo no在一个事务中是从0开始递增的，也就是说只要事务没提交，每生成一条undo日志，那么该条日志的undo no就增1。
    • redo是以组mtr为单位记录，复写的；
    • undo日志是以事务为单位记录、还原的？
  • undo日志也有很多种类。

行记录

trx隐藏列

行格式中有一个trx_id字段，表示现在对该行进行修改的事务id。

roll_pointer隐藏列

指向记录对应的undo日志的一个指针。

undo日志类型

insert类


只需要记录插入id即可，回滚时根据id删除。

delete类

  • 在对一条记录进行delete mark操作前，需要把该记录的旧的trx_id和roll_pointer隐藏列的值都给记到对应的undo日志中来，形成版本链：


  • 额外记录删除时记录各列数据和索引各列信息。

删除页的过程：
1）将记录的delete_mask标识为改为1

2）删除事务提交后，有专门线程来将delete标识位为1的记录回收

update类

不更新主键
  • 如果更新后的列和更新前的列占用的存储空间都一样大，那么就可以进行就地更新。
  • 否则，用户线程直接先删除记录（记录加入垃圾链表），然后插入新记录。

更新主键
  • 将旧记录进行delete mark操作
  • 创建新记录并插入

对应主键更新，可以delete mask，否则必须先硬删除记录再插入新记录。

undo页面

FIL_PAGE_UNDO_LOG页面，该页面专门存储undo日志

  • TRX_UNDO_PAGE_TYPE：本页面准备存储什么种类的undo日志。
  • TRX_UNDO_PAGE_START：表示在当前页面第一条undo日志的起始偏移量。
  • TRX_UNDO_PAGE_FREE：最后一条undo日志结束时的偏移量。
  • TRX_UNDO_PAGE_NODE：代表一个List Node结构，连接前后的Undo页面（就像INODE页面一样）。

Undo页面链表

通过TRX_UNDO_PAGE_NODE形成链表。

一个undo页只记录一种类型的undo日志，不同类型的undo页形成链表：

针对普通表和临时表，又分别形成两组Undo日志。

这些链表是按需分配的。

undo链表是属于事务的：

每个事务分两组（普通/临时），每组有两条链表（update/insert），每个链表由若干该类型的undo页面连接构成。

undo日志具体写入过程

undo log segment header

每一个Undo链表都对应着一个段，称之为Undo Log Segment。

对于数据页，每个索引分两个段（叶子段/非叶子段），每个段有若干页面。
对于undo页，每个链表就是一个段。

  • TRX_UNDO_STATE：本Undo链表处在什么状态。
    • TRX_UNDO_ACTIVE：活跃状态，也就是一个活跃的事务正在往这个段里边写入undo日志。
    • TRX_UNDO_CACHED：被缓存的状态，等待被重用
    • TRX_UNDO_TO_FREE：insert undo链表不能被重用
    • TRX_UNDO_TO_PURGE：update undo链表不能被重用
    • 处于PREPARE阶段的事务产生的undo日志。
  • TRX_UNDO_LAST_LOG：本Undo页面链表中最后一个Undo Log Header的位置。
  • TRX_UNDO_FSEG_HEADER：本Undo页面链表对应的段的Segment Header信息
  • TRX_UNDO_PAGE_LIST：Undo页面链表的基节点。
  • undo page header 主要记录本页的指针信息
  • undo log segment 主要记录本undo log页链表的信息，便于快速搜索。
  • Segment Header 可以用来快速定位本段的INODE Entry（INODE Entry用来管理一个段中各页面的链表）。

undo log header

定义：同一个事务向一个Undo页面链表中写入的undo日志算是一个组，在undo log header记录这些组信息。

小结

  • 链表的第一个页面，会填充Undo Page Header、Undo Log Segment Header、Undo Log Header这3个部分，之后才开始正式写入undo日志。
  • 对于其他的页面只会填充Undo Page Header。

如前面总结，一个事务会持有多个undo log页面链表，在第一个链表中放入：

  • undo log segment header：记录本链表的状态（写入中？），并定位到INODE Entry，便于页面管理。
  • undo log header：记录本链表与事务相关的记录。

重用Undo页面

Undo页面链表是否可以被重用的条件很简单：

  • 该链表中只包含一个Undo页面。
  • 该Undo页面已经使用的空间小于整个页面空间的3/4：
    • insert undo链表可立刻删除。
    • update undo链表不能立刻删除。


回滚段

Rollback Segment Header页面

在这个页面中存放了各个Undo页面链表的frist undo page的页号，他们把这些页号称之为undo slot。

每个链表的头部记录了链表的状态信息和事务信息
这个页面中通过引用各个链表头部对链表进行统一的管理，方便复用：

  • 事务从回滚段中申请Undo链表。
  • 事务提交后，符合重用条件的可复用，不可重用的根据类型区分：
  • insert类型可释放
  • update类型需要放入history链表中。

多个回滚段

InnoDB中有127个回滚段，一个回滚段管理1024个undo链表。

回滚段的分类

  • 普通表回滚段
  • 临时表回滚段

划分原因：在修改针对普通表的回滚段中的Undo页面时，需要记录对应的redo日志，而修改针对临时表的回滚段中的Undo页面时，不需要记录对应的redo日志。

回滚段相关配置

  • 配置回滚段数量：启动参数innodb_rollback_segments
  • 通过innodb_undo_directory指定undo表空间所在的目录
  • 通过innodb_undo_tablespaces定义undo表空间的数量

总结

  • 一个事务对应多个mtr，一个mtr对应多个undo日志
  • undo日志存在undo页中，undo页分update和insert类。
  • 一个undo页不够记录一个事务的所有操作，所以要形成链表。每个事务持有2部分链表（普通/临时），1部分链表包括2条链表（update/insert）链表。（insert类型的undo页面在事务提交后就可以释放，update则不可以，这是处于版本链管理的需要）
  • 一个链表就是一个undo页面段，其也有一个INODE结构来管理这个段中各个页面?
  • 回滚段页面有各个链表头，实现对链表的管理。

版本链

注意insert操作提交后对应的undo日志就没用了。

每次对记录进行改动，都会记录一条undo日志，每条undo日志也都有一个roll_pointer属性：

ReadView

对于使用READ COMMITTED和REPEATABLE READ隔离级别的事务来说：
必须保证可以读到已经提交了的事务修改过的记录。

事务为了保证这种一致性，会在对应的时刻生成一个ReadView对象：

  • m_ids：表示在生成ReadView时当前系统中活跃的读写事务的事务id列表。
  • min_trx_id：表示在生成ReadView时当前系统中活跃的读写事务中最小的事务id，也就是m_ids中的最小值。
  • max_trx_id：表示生成ReadView时系统中应该分配给下一个事务的id值。
  • creator_trx_id：表示生成该ReadView的事务的事务id。

这样事务可以根据被访问版本的trx_id值判断：

  • 小于min_trx_id，说明trx_id事务已经被提交，可访问
  • 大于max_trx_id，说明trx_id事务在本事务开启后开启的，不可访问
  • 等于creator_trx_id，说明是本事务进行的修改，可访问
  • min与max之间，判断trx_id是否在m_ids中：若在则说明创建ReadView时trx_id事务未提交，不可访问，否则说明trx_id事务已经提交，可访问。

使用

Meta-Data Locks

使用
一般不需要显示的使用 MDL（元数据锁），因为对数据库表进行操作时，会自动给这个表加上 MDL：

  • 对一张表进行 CRUD 操作时，加的是MDL 读锁；
  • 对一张表做结构变更操作的时候，加的是MDL 写锁；
  • 事务提交后才会释放，这意味着事务执行期间，MDL 是一直持有的。

范围与目的
MDL 是为了保证当用户对表执行 CRUD 操作时，防止其他线程对这个表结构做了变更。同理，对表结构进行操作时，阻塞CRUD操作。

IS/IX Locks

使用
意向锁是一种表锁，在对某表的记录加锁前先对该表加上意向锁。

范围与目的
意向锁是一种表锁，在对某表的记录加锁前先对该表加上意向锁，想加表锁的事务可以通过查看本表有没有意向锁判断是否有其他事务已经加了行锁。

  • 意向共享锁，英文名：Intention Shared Lock，简称IS锁。当事务准备在某条记录上加S锁时，需要先在表级别加一个IS锁。
  • 意向独占锁，英文名：Intention Exclusive Lock，简称IX锁。当事务准备在某条记录上加X锁时，需要先在表级别加一个IX锁。

AUTO-INC Locks

使用
主键设置为AUTO-INC后，插入记录时会自动使用AUTO-INT锁。

范围与目的
AUTO-INC 锁是特殊的表锁机制，锁不是再一个事务提交后才释放，而是再执行完插入语句后就会立即释放。在插入数据时，会加一个表级别的 AUTO-INC 锁，然后为被 AUTO_INCREMENT 修饰的字段赋值递增的值，等插入语句执行完成后，才会把 AUTO-INC 锁释放掉。

优化

  • 问题：AUTO-INC 锁在对大量数据进行插入的时候，会影响插入性能，因为另一个事务中的插入会被阻塞。
  • 解决：在 MySQL 5.1.22 版本开始，InnoDB 存储引擎提供了一种轻量级的锁来实现自增，在插入数据的时候，会为被 AUTO_INCREMENT 修饰的字段加上轻量级锁，然后给该字段赋值一个自增的值，就把这个轻量级锁释放了，而不需要等待整个插入语句执行完后才释放锁。
  • 配置方式： 使用innodb_autoinc_lock_mode系统变量进行配置：
    • 0：为默认AUTO-INC锁
    • 1：普通 insert 语句为轻量级锁，类似 insert … select 这样的批量插入数据的语句为AUTO-INC锁。
    • 2：为轻量级锁，申请自增主键后就释放锁，并不需要等语句执行后才释放。
  • 新的问题：当 innodb_autoinc_lock_mode = 2 是性能最高的方式，但是当搭配 binlog 的日志格式 statement 一起使用的时候，在「主从复制的场景」中会发生数据不一致的问题。因为binlog只会记录原始语句，且从库在执行时会一个session事务一个session事务的执行，即不会像主库需要处理两个session的并发操作，这会导致从库id与主库id不一致。
  • 新的解决：将binlog的日志格式配置为row即可。

无锁

Record Locks

读操作：当前读，分为S锁和X锁：

写操作：

  • INSERT：不加锁，但使用隐式锁保证新增记录在提交前无法被访问。
  • DELETE：X锁
  • UPDATE：
    • 原地更新字段时（长度无变化）：X锁
    • 删除新增字段时（长度有变化）：先后执行DELETE和INSERT。
    • 更新主键时：先后执行DELETE和INSERT。
Gap Locks

只存在于可重复读隔离级别，目的是为了解决可重复读隔离级别下幻读的现象，加锁后其他事务无法在锁记录前面的区间(3，8)插入新记录。

通过将锁放置在页面的最大记录上，即锁住了区间(20,+∞)，其他事务就不能插入新记录了。

间隙锁虽然存在 X 型间隙锁和 S 型间隙锁，但是并没有什么区别，间隙锁之间是兼容的，即两个事务可以同时持有包含共同间隙范围的间隙锁，并不存在互斥关系，因为间隙锁的目的是防止插入幻影记录而提出的。

Next-Key Locks

相当于Record Locks+Gap Locks，锁住了区间(3,8]

next-key lock 是包含记录锁的，所以如果一个事务获取了 X 型的 next-key lock，那么另外一个事务在获取相同范围的 X 型的 next-key lock 时是会被阻塞的。

Insert-Intention Locks

Insert Intention Locks：插入意向锁，事务在插入时被gap lock或者next-key lock阻塞，则生成一个Insert Intention Locks锁，其主要起唤醒作用。


隐式锁

如果某事务A插入记录时，另一个事务B要读取这个记录，会产生冲突（此时初始化未完成）。

  • 对于聚簇索引，事务B发现记录的trx_id（即事务A）处于活跃状态时，会为事务A创建一个X锁，并为自己创建一个X锁，等待事务A释放X锁后，事务B即可访问新插入的数据。
  • 对于二级索引，由于二级索引的记录不包含事务ID，因此通过Page Header中的PAGE_MAX_TRX_ID属性判断最后修改该页面的事务是否是已提交状态（与当前活跃事务的最小id对比），如果均已提交，则说明不存在冲突，否则需要回到聚簇索引进行判断。

锁的使用规则

锁类型的使用场景

  • 在读已提交隔离级别下，行级锁的种类只有记录锁
  • 在可重复读隔离级别下，行级锁的种类除了有记录锁，还有间隙锁以及Next Key锁。

锁的获取与释放

在 MySQL 中，行锁的获取和释放是自动进行的。当我们执行一条语句时，MySQL 会根据需要自动获取行锁，并在事务结束或者提交时自动释放行锁。行锁的获取和释放是基于事务的，只有在事务中操作的行才会被锁定。

RR模式下的锁退化

加锁的对象是索引，RR模式下加锁的基本单位是 next-key lock。
next-key lock是由记录锁和间隙锁组合而成的，next-key lock 是前开后闭区间，而间隙锁是前开后开区间。但是，next-key lock 在一些场景下会退化成记录锁或间隙锁：在能使用记录锁或者间隙锁就能避免幻读现象的场景下， next-key lock 就会退化成记录锁或间隙锁。

唯一索引等值查询
  • 当查询的记录是「存在」的，在索引树上定位到这一条记录后，将该记录的索引中的 next-key lock 会退化成记录锁。
  • 当查询的记录是「不存在」的，在索引树找到第一条大于该查询记录的记录后，将该记录的索引中的 next-key lock 会退化成间隙锁。（部分解决幻读问题，查询不到某记录时，不允许其他事务再插入该记录）

例子：表中id有1,5,6时，查询id = 2的记录时，会锁住(1,5)开区间。

唯一索引范围查询

当唯一索引进行范围查询时，会对每一个扫描到的索引加 next-key 锁：

  • 情况一：针对「大于」的范围查询：
    • 命中大于条件的索引（包括最大索引）加 next-key 锁。
  • 情况二：针对「大于等于」的范围查询：
    • 命中大于条件的索引（包括最大索引）加 next-key 锁。
    • 命中等于条件的索引的 next-key 锁会退化成记录锁。
  • 情况三：针对「小于」的范围查询：
    • 命中小于条件/等于条件的索引加 next-key 锁。
    • 恰好大于该条件的第一个索引或最大索引加间隙锁。
  • 情况四：针对「小于等于」的范围查询：
    • 命中小于条件的索引加 next-key 锁。
    • 若命中等于条件的索引存在，对其加 next-key 锁。（这是因为主键的唯一性，在该索引上加next-key即满足要求）；若不存在，则恰好大于该条件的第一个索引或最大索引加间隙锁。

非唯一索引等值查询
  • 当查询的记录是「存在」时：
    • 命中等于条件的二级索引加 next-key 锁。
    • 恰好大于该条件的第一个二级索引或最大二级索引加间隙锁。
    • 对命中的记录的主键索引加 记录锁。
  • 当查询到的记录「不存在」时：
    • 恰好大于该条件的第一个二级索引或最大二级索引加间隙锁。
非唯一索引范围查询

非唯一索引范围查询，索引的 next-key lock 不会有退化为间隙锁和记录锁的情况，即：

  • 命中查询条件的二级索引（包括最大索引）加 next-key 锁。
  • 对命中的记录的主键索引加 记录锁。

读锁如果不涉及回表操作，不会对聚簇索引进行加锁：

  • for update 加的是写锁，写锁默认认为会对数据做更改，不管查询有没有涉及到回表都会对聚簇索引(主键索引)加锁；
  • lock in share model 加的是读锁，如果没有涉及到回表（像覆盖索引），不会对聚簇索引(主键索引)加锁。
没有加索引的查询

如果锁定读查询语句，没有使用索引列作为查询条件，或者查询语句没有走索引查询，导致扫描是全表扫描。那么，每一条记录的索引上都会加 next-key 锁，这样就相当于锁住的全表，这时如果其他事务对该表进行增、删、改操作的时候，都会被阻塞。

在线上在执行 update、delete、select … for update 等具有加锁性质的语句，一定要检查语句是否走了索引，如果是全表扫描的话，会对每一个索引加 next-key 锁，相当于把整个表锁住了。
虽然这看起来效果和表锁类似，但其本质上还是若干行锁的叠加，不是真正的表锁（有文章说未命中索引会加表锁，结果测试，发现加的是行锁）

如何避免锁全表

问题
在 update、delete、select … for update 语句的 where 条件没有使用索引，就会全表扫描。即使使用了索引，也不一定会走索引，其关键还得看这条语句在执行过程中，优化器最终选择的是索引扫描，还是全表扫描，如果走了全表扫描，就会对全表的记录加锁了。

解决

  1. 将 MySQL 里的 sql_safe_updates 参数设置为 1，
    • update 语句必须满足如下条件之一才能执行成功：
      • 使用 where，并且 where 条件中必须有索引列；
      • 使用 limit；
      • 同时使用 where 和 limit，此时 where 条件中可以没有索引列；
    • delete 语句必须满足以下条件能执行成功：
      • 使用 where，并且 where 条件中必须有索引列；
      • 同时使用 where 和 limit，此时 where 条件中可以没有索引列；
  2. 如果 where 条件带上了索引列，但是优化器最终扫描选择的是全表，而不是索引的话，我们可以使用 force index([index_name]) 可以告诉优化器使用哪个索引，以此避免有几率锁全表带来的隐患。

死锁

死锁检测
当并发系统中不同线程出现循环资源依赖，涉及的线程都在等待别的线程释放资源时，就会导致这几个线程都进入无限等待的状态，称为死锁。可以通过下面的指令查看死锁show engine innodb status

死锁解除

  • 超时等待，事物超时自动回滚(innodb_lock_wait_timeout 默认50s)
  • 主动死锁检测，事物请求锁的时候采用 wait-for graph 等待图的方式进行死锁检测（innodb_deadlock_detect 默认on）
  • 发现死锁也可以人为 kill 进程

各种锁发生死锁的情况

  • 表级锁：开销小，加锁快；不会出现死锁；锁定粒度大，发生锁冲突的概率最高,并发度最低。
  • 行级锁：开销大，加锁慢；会出现死锁；锁定粒度最小，发生锁冲突的概率最低,并发度也最高。
  • 页面锁：开销和加锁时间界于表锁和行锁之间；会出现死锁；锁定粒度界于表锁和行锁之间，并发度一般

InnoDB锁结构

  • 事务信息：事务的指针
  • 索引信息：加锁记录所属的索引
  • 表锁／行锁信息：
    • 标记锁了哪个表：SpaceID
    • 哪个表哪行记录：SpaceID,PageNumber,n_bits(一个数量，以便后面通过bit位标记页内哪些行加了锁)
  • type_mode：这是一个32位的数，被分成了lock_mode、lock_type和rec_lock_type三个部分，如图所示：
    • 锁的模式（lock_mode），占用低4位，可选的值如下：
      • LOCK_IS（十进制的0）：表示共享意向锁，也就是IS锁。
      • LOCK_IX（十进制的1）：表示独占意向锁，也就是IX锁。
      • LOCK_S（十进制的2）：表示共享锁，也就是S锁。
      • LOCK_X（十进制的3）：表示独占锁，也就是X锁。
      • LOCK_AUTO_INC（十进制的4）：表示AUTO-INC锁。
    • 锁的类型（lock_type），占用第5～8位，不过现阶段只有第5位和第6位被使用：
      • LOCK_TABLE（十进制的16），也就是当第5个比特位置为1时，表示表级锁。
      • LOCK_REC（十进制的32），也就是当第6个比特位置为1时，表示行级锁。
    • 行锁的具体类型（rec_lock_type），使用其余的位来表示。只有在lock_type的值为LOCK_REC时，也就是只有在该锁为行级锁时，才会被细分为更多的类型：
      • LOCK_ORDINARY（十进制的0）：表示next-key锁。
      • LOCK_GAP（十进制的512）：也就是当第10个比特位置为1时，表示gap锁。
      • LOCK_REC_NOT_GAP（十进制的1024）：也就是当第11个比特位置为1时，表示正经记录锁。
      • LOCK_INSERT_INTENTION（十进制的2048）：也就是当第12个比特位置为1时，表示插入意向锁。
      • LOCK_WAIT（十进制的256） ：也就是当第9个比特位置为1时，表示is_waiting为true，也就是当前事务尚未获取到锁，处在等待状态；当这个比特位为0时，表示is_waiting为false，也就是当前事务获取锁成功。

  • 其他信息：为了更好的管理系统运行过程中生成的各种锁结构而设计了各种哈希表和链表。
  • 一堆比特位：数量由行锁信息的n_bits属性决定，每bit与页内记录的heap_no对应

并发控制技术

从宽泛意义上讲，目前有三种并发控制技术：

  • 悲观并发控制(PCC)：心态悲观，假定多用户并发的事物在处理时都会引起并发冲突，每次操作数据的时候都会上锁。先取锁再访问的策略，为数据的安全提供了保证，但是加锁会产生额外的开销，增加死锁的机会，只读型事物不会产生冲突也不需要加锁。
  • 乐观并发(OCC)：心态乐观，假定多用户并发的事物在处理时不会彼此互相影响，只在提交时检查有没有其它事物修改了该数据可以获得更大的吞吐量，但是发生冲突事物就会回滚重新执行。
  • 多版本并发(MVCC)：每个写操作都会创建一个新版本的数据，读操作根据可见性规则返回其中一个数据快照读 - 写冲突不加锁，非阻塞读的同时避免了脏读和不可重复读，但需要管理和挑选数据版本。

注意：

  • MVCC不是与悲观、乐观并发控制对立的，很直观的一点MVCC可以在不加锁的情况下解决读-写冲突，并不能解决写-写冲突，写操作还是需要上锁
  • MVCC可以与悲观并发或乐观并发结合使用来提高并发的性能

事务隔离级别实现

事务隔离级别，其本质是并发事务之间可见性的等级，其重点在于多次读的结果是否一致：读行为会根据隔离级别有不同的实现。

MVCC和锁则是实现事务隔离级别的手段：

  • 普通读：InnoDB通过MVCC实现了RC和RR，其通过读版本链中的数据避免对记录加锁产生的消耗（本质上是弱一致性读，也叫快照读，写操作依然需要一致性读）。
  • 当前读：通过锁阻止其他事务对记录的更改（保证一致性，随着隔离级别的升高，使用的锁也不同）
  • 写操作：除了Serializable，所有隔离级别都会上表级IX锁和行级记录X锁，事务提交后才释放

Read Uncommitted

实现
普通读时：无锁快照读，读最新值
性能
性能最好，但不会比其他级别好很多。
实用性
若事务回滚，其他事务会读到错误的信息，因此除非真的有必要，否则很少使用。

Read Commited

实现

  • 普通读时：通过MVCC方式解决脏读，在事务中每一个查询语句执行时生成一个ReadView。
  • 当前读时：通过记录锁解决了脏读&不可重复读，此时其他事务要修改该记录会失败。

性能
性能较好，半一致性读特性增加了update操作的并发性。

半一致性读
semi-consistent read是read committed与consistent read两者的结合：

  • 原理：执行update语句时：如果读到一行已经加锁的记录（可能其他事务正在更新），此时InnoDB返回记录最近提交的版本，由MySQL上层判断此版本是否满足update的where条件：
    • 若满足，则MySQL会重新发起一次读操作，此时会读取该行的最新版本并加锁等待。
    • 若不满足，说明该记录加锁前就不满足条件，此时不会加锁。
  • 优点：减少同一行记录的锁冲突及锁等待；
  • 条件：半一致读只会发生在RC隔离级别下，或者是参数innodb_locks_unsafe_for_binlog被设置为true时。
  • 其他：innodb_locks_unsafe_for_binlog 参数在 8.0 版本中已被去除（可见，这是一个可能会导致数据不一致的参数，官方也不建议使用了）。

半一致性读基于一种假设：当前被加锁的记录如果之前不符合where条件，之后应该也不符合，所以对其进行了忽略。但该假设并不总是成立，在RC下，需要业务理解好半一致读的特性避免漏更新（即本质上两个事务按照不同的顺序执行时结果不一样）。因此也要避免使用statement模式的binlog，其会导致主从不一致。

实用性
实用性较好，是大多数数据库的默认隔离级别。

Repeatable Read

实现

  • 普通读时：通过MVCC方法解决了脏读、不可重复读，在事务中的第一个查询语句（事务真正开启的时点）时会生成一个ReadView。
  • 当前读时：通过 next-key lock（记录锁+间隙锁） 方式解决了不可重复读&大部分幻读，此时其他事务要在间隙锁的范围内插入记录会失败。

性能
当前读时存在间隙锁，导致出现死锁的几率比RC大的多，且未命中索引会锁表！

未完全解决的幻读
当某个事务先对一个不存在的记录做普通读，后对该记录（已经被其他事务插入并提交）做当前读时，则会出现幻读。

实用性
可以解决幻读，但性能没有RC好，是MySQL（InnoDB）的默认数据库事务级别。

Serializable

实现
不再支持快照读，所有普通select语句都会被隐式的转化为select ... in share mode，保证读写操作串行化运行，从而达到该级别的要求。

实用性
并发性能较低，实用性较差

隔离级别比较

为什么大多数企业应用使用RC而不是RR？

  • 在RR隔离级别下，会产生间隙锁，这样在频繁的插入、更新过程中更容易产生锁等待。
  • 在RR隔离级别下，条件列未命中索引会锁表！而在RC隔离级别下，只锁行！
  • 在RC隔离级别下，半一致性读(semi-consistent)特性增加了update操作的并发性！

在真实业务场景下，幻读很少会成为影响业务的问题，如果一致性要求没那么强，可以使用提交读以提高并发性。

为什么Mysql默认使用RR？
binlog有几种格式：

  • statement：记录的是修改SQL语句
  • row：记录的是每行实际数据的变更
  • mixed：statement和row模式的混合

RC 隔离级别不支持 statement 格式的bin log，该格式的复制会导致主从数据的不一致，这也是为什么MySQL默认使用RR隔离级别的原因。复制时最好使用：binlog_format=row。

六、bin log

redo log与bin log一致性

橙色的流程在InnoDB内部执行，蓝色的部分在MySQL Server层的执行器中执行。

  • Point A：如果服务器异常关闭发生在Point A以及之前的时间点，这个时候redolog 和 binlog都没有任何记录，事务还未提交，不会造成任何影响。
  • Point B：当服务器启动的时候发现redo log里处于prepare状态的记录，这个时候需要检查binlog是否完整包含此条redo log的更新内容（通过全局事务ID对应），发现binlog中还未包含此事务变更，则丢弃此次变更。
  • Point C：和Point B基本相同，只不过此时发现binlog中包含redo log的更新内容，此时事务会进行提交。
  • Point D：binlog中和数据库中均含有此事务的变更，没有任何影响。
